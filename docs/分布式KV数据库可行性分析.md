分布式 KV 数据库可行性分析                                                                                                                                                                                                                  
                                                                                                                                                                                                                                              
  一、当前架构盘点                                                                                                                                                                                                                            
                                                                                                                                                                                                                                              
  ┌──────────┬───────────────────────────────────────────────────────────────────────┬──────────────────┐                                                                                                                                     
  │ 层次     │ 已有能力                                                              │ 缺失部分         │                                                                                                                                     
  ├──────────┼───────────────────────────────────────────────────────────────────────┼──────────────────┤                                                                                                                                     
  │ 存储引擎 │ LSM-Tree、MemTable、WAL、SSTable、Compaction、BloomFilter、BlockCache │ 分区数据独立管理 │                                                                                                                                     
  ├──────────┼───────────────────────────────────────────────────────────────────────┼──────────────────┤                                                                                                                                     
  │ 复 制    │ 主从复制、全量同步(RDB)、部分重同步(PSYNC)、Backlog、心跳保活         │ 多主/共识复制    │                                                                                                                                     
  ├──────────┼───────────────────────────────────────────────────────────────────────┼──────────────────┤                                                                                                                                     
  │ 网络协议 │ RESP (Redis 兼容)、epoll/kqueue 事件循环                              │ 节点间内部协议   │                                                                                                                                     
  ├──────────┼───────────────────────────────────────────────────────────────────────┼──────────────────┤                                                                                                                                     
  │ 客户端   │ Python/Go/Node.js SDK                                                 │ 集群路由感知     │                                                                                                                                     
  ├──────────┼───────────────────────────────────────────────────────────────────────┼──────────────────┤                                                                                                                                     
  │ 事 务    │ 单节点 snapshot 事务                                                  │ 分布式事务       │                                                                                                                                     
  └──────────┴───────────────────────────────────────────────────────────────────────┴──────────────────┘                                                                                                                                     
                                                                                                                                                                                                                                              
  二、可以实现什么 — 三种演进路线                                                                                                                                                                                                             
  🅰 简单路线：客户端分片（无中心）                                                                                                                                                                                                            
                                                                                                                                                                                                                                              
  原理：不在服务端改任何代码，只扩展客户端 SDK。客户端用一致性哈希决定 key → node 映射。                                                                                                                                                      
                                                                                                                                                                                                                                              
  改动量：约 300-500 行（客户端层），服务端零改动。                                                                                                                                                                                           
                                                                                                                                                                                                                                              
  优缺点：                                                                                                                                                                                                                                    
  •✅ 不改服务端，利用现有主从复制获得高可用                                                                                                                                                                                                  
  •✅ 线性扩展写吞吐                                                                                                                                                                                                                          
  •❌ 节点增减需要客户端感知、手动调整哈希环                                                                                                                                                                                                  
  •❌ 跨节点操作（MSet、Rename）无法原子化                                                                                                                                                                                                    
  •❌ 无自动 rebalance                                                                                                                                                                                                                        
                                                                                                                                                                                                                                              
  适用场景：缓存层、写入密集型、接受最终一致性的场景。                                                                                                                                                                                        
  🅱 中等路线：Proxy 协调层                                                                                                                                                                                                                    
                                                                                                                                                                                                                                              
  原理：在 Server 旁加一个 Proxy 层（无状态），解析 RESP 命令，根据 key 哈希转发到后端存储节点。后端仍是单个 LightKV server（利用已有主从复制）。                                                                                             
                                                                                                                                                                                                                                              
    Client → Proxy (一致性哈希) → LightKV Node 1                                                                                                                                                                                              
                                 → LightKV Node 2                                                                                                                                                                                             
                                 → LightKV Node 3                                                                                                                                                                                             
                                       ↓                                                                                                                                                                                                      
                                 各自的 Slave 做高可用                                                                                                                                                                                        
                                                                                                                                                                                                                                              
  改动量：                                                                                                                                                                                                                                    
  •新增 proxy/ 目录 ≈ 1500-2000 行 C++                                                                                                                                                                                                        
  •Proxy 解析 RESP → 查找路由 → 转发 → 聚合响应                                                                                                                                                                                               
  •服务端仅需加几个运维命令（CLUSTER NODES、CLUSTER SLOTS）                                                                                                                                                                                   
  •利用已有主从复制做 failover                                                                                                                                                                                                                
                                                                                                                                                                                                                                              
  优缺点：                                                                                                                                                                                                                                    
  •✅ 客户端无感知（连 Proxy 就像连 Redis Cluster）                                                                                                                                                                                           
  •✅ Proxy 无状态，可水平扩展多个 Proxy                                                                                                                                                                                                      
  •✅ 复用主从复制做高可用                                                                                                                                                                                                                    
  •❌ Proxy 是瓶颈（需 epoll 高性能转发）                                                                                                                                                                                                     
  •❌ 跨 slot 多键操作受限                                                                                                                                                                                                                    
  •❌ 节点增减仍需手动触发 slot 迁移                                                                                                                                                                                                          
                                                                                                                                                                                                                                              
  适用场景：需要 Redis Cluster 风格的无缝水平扩展。                                                                                                                                                                                           
  🅲 最终路线：Raft 共识 + 自动分片                                                                                                                                                                                                            
                                                                                                                                                                                                                                              
  原理：引入 Raft 共识层做元数据管理（控制面），数据面分片并支持自动 rebalance。                                                                                                                                                              
                                                                                                                                                                                                                                              
                        ┌─────────────┐                                                                                                                                                                                                       
                        │ Raft 控制面  │  (3-5 节点, 存 slot 映射)                                                                                                                                                                            
                        │  (元数据)    │                                                                                                                                                                                                      
                        └──────┬──────┘                                                                                                                                                                                                       
                               │                                                                                                                                                                                                              
                ┌──────────────┼──────────────┐                                                                                                                                                                                               
                ▼              ▼              ▼                                                                                                                                                                                               
           ┌─────────┐   ┌─────────┐   ┌─────────┐                                                                                                                                                                                            
           │Shard 1  │   │Shard 2  │   │Shard 3  │                                                                                                                                                                                            
           │(Raft 3) │   │(Raft 3) │   │(Raft 3) │                                                                                                                                                                                            
           └─────────┘   └─────────┘   └─────────┘                                                                                                                                                                                            
                                                                                                                                                                                                                                              
  改动量：约 8000-15000 行 C++，核心组件：                                                                                                                                                                                                    
  1.Raft 共识库（或集成 nuRaft / 自研）                                                                                                                                                                                                       
  2.Cluster 状态机 — slot 分配、节点上下线                                                                                                                                                                                                    
  3.Slot 迁移引擎 — 异步数据搬迁                                                                                                                                                                                                              
  4.分布式事务 — 2PC 或 TCC                                                                                                                                                                                                                   
  5.Gossip 协议 — 节点发现、故障检测                                                                                                                                                                                                          
                                                                                                                                                                                                                                              
  优缺点：                                                                                                                                                                                                                                    
  •✅ 强一致性、自动 failover、自动 rebalance                                                                                                                                                                                                 
  •✅ 真正的分布式 KV                                                                                                                                                                                                                         
  •❌ 工程量大，复杂度高                                                                                                                                                                                                                      
  •❌ Raft 写性能有天花板（Leader 瓶颈）                                                                                                                                                                                                      
  •❌ 与现有主从复制架构重叠（需要重写复制层）                                                                                                                                                                                                
  三、推荐路线                                                                                                                                                                                                                                
                                                                                                                                                                                                                                              
  建议分 3 阶段演进：                                                                                                                                                                                                                         
                                                                                                                                                                                                                                              
  ┌─────────┬────────────────────────────────────────────────────┬──────────┬────────────────┐                                                                                                                                                
  │ 阶段    │ 内容                                               │ 工期估计 │ 并行度         │                                                                                                                                                
  ├─────────┼────────────────────────────────────────────────────┼──────────┼────────────────┤                                                                                                                                                
  │ Phase 1 │ 客户端分片（一致性哈希） + 客户端自动故障转移      │ 3-5 天   │ SDK 独立开发   │                                                                                                                                                
  ├─────────┼────────────────────────────────────────────────────┼──────────┼────────────────┤                                                                                                                                                
  │ Phase 2 │ Proxy 层 + SLOTS 路由 + 基于现有主从的自动故障切换 │ 2-3 周   │ Proxy 独立进程 │                                                                                                                                                
  ├─────────┼────────────────────────────────────────────────────┼──────────┼────────────────┤                                                                                                                                                
  │ Phase 3 │ 元信息持久化（Raft 控制面）+ slot 自动迁移         │ 4-6 周   │ 需设计共识层   │                                                                                                                                                
  └─────────┴────────────────────────────────────────────────────┴──────────┴────────────────┘                                                                                                                                                
                                                                                                                                                                                                                                              
  推荐优先做 Phase 1 + Phase 2                                                                                                                                                                                                                
                                                                                                                                                                                                                                              
  因为：                                                                                                                                                                                                                                      
  1.Phase 1 几乎零服务端改动，能快速让用户用上多节点                                                                                                                                                                                          
  2.Phase 2 的 Proxy 是一个独立的无状态进程，不侵入现有 server.cpp 的复杂事件循环                                                                                                                                                             
  3.主从复制已可用 — 每个分片内部部署 1 Master + N Slave 即得高可用                                                                                                                                                                           
  4.Proxy 可以和 Client SDK 共存，客户端可以连 Proxy 或直连单节点，向后兼容

  五、关键风险                                                                                                                                                                                                                                
                                                                                                                                                                                                                                              
  ┌──────────────────────────────────────────────────────────────┬─────────────────────────────────────────┐                                                                                                                                  
  │ 风险                                                         │ 缓解措施                                │                                                                                                                                  
  ├──────────────────────────────────────────────────────────────┼─────────────────────────────────────────┤                                                                                                                                  
  │ 现有 Server 是单线程事件循环，写操作处理完后直接 Feed 复制流 │ Proxy 转发只需转发 RESP，不需改复制路径 │                                                                                                                                  
  ├──────────────────────────────────────────────────────────────┼─────────────────────────────────────────┤                                                                                                                                  
  │ Server 不支持 CLUSTER 命令                                   │ 加少量命令处理分支                      │                                                                                                                                  
  ├──────────────────────────────────────────────────────────────┼─────────────────────────────────────────┤                                                                                                                                  
  │ 没有节点间 RPC                                               │ Proxy 复用现有 RESP 协议与后端通信      │                                                                                                                                  
  ├──────────────────────────────────────────────────────────────┼─────────────────────────────────────────┤                                                                                                                                  
  │ 跨分片事务                                                   │ Phase 1/2 不支持，Phase 3 可考虑        │                                                                                                                                  
  └──────────────────────────────────────────────────────────────┴─────────────────────────────────────────┘                                                                                                                                  
                                                                                                                                                                                                                                              
  结论：在当前的架构上实现分布式 KV 是可行的，且投入产出比最高的路径是「Phase 1 客户端分片 + Phase 2 Proxy 层」。核心存储引擎、复制协议、网络层都不需要大改，增量开发集中在「路由」这一层。

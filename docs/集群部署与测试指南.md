## 1. 环境要求

### 1.1 硬件

| 项目 | 最低要求 | 推荐配置 |
|------|---------|---------|
| CPU | 2 核 | 4 核+ |
| 内存 | 2GB | 8GB+ |
| 磁盘 | 10GB | 50GB+ SSD |
| 网络 | 千兆 | 万兆 |

### 1.2 软件

| 组件 | 版本要求 |
|------|---------|
| 操作系统 | Linux 5.x+ |
| 编译器 | GCC 10+ / Clang 12+（C++20） |
| CMake | 3.16+ |
| 可选 | `nc` / `redis-cli`（用于命令行验证）|

### 1.3 节点间网络

三节点集群需要以下端口互通：

| 端口 | 用途 | 说明 |
|------|------|------|
| `6379` | RESP 服务端口 | 客户端连接、数据读写 |
| `16379` | Raft 内部通信端口 | 选举、日志复制、快照传输 |

确保防火墙放行上述端口：
```bash
# Ubuntu / Debian
sudo ufw allow 6379/tcp
sudo ufw allow 16379/tcp

# CentOS / RHEL
sudo firewall-cmd --add-port=6379/tcp --permanent
sudo firewall-cmd --add-port=16379/tcp --permanent
sudo firewall-cmd --reload
```

## 2. 编译构建

### 2.1 获取源码

```bash
git clone https://github.com/your-org/lightkv.git
cd lightkv
```

### 2.2 Release 构建

```bash
mkdir build && cd build
cmake .. -DCMAKE_BUILD_TYPE=Release -DLIGHTKV_BUILD_TESTS=OFF -DLIGHTKV_BUILD_SERVER=ON
make -j$(nproc) lightkv_server
```

> **注意**：Release 构建默认使用 `-O3` 优化。如果需要在不同 CPU 架构的机器间分发二进制，确保 `-march=native` 已被移除（v2.0+ 默认已移除）。

### 2.3 同步到其他节点

```bash
# 将二进制同步到其他节点
scp build/lightkv_server user@node2:~/lightkv/build/
scp build/lightkv_server user@node3:~/lightkv/build/
```

## 3. 三节点集群部署

### 3.1 节点规划

| 节点 | IP | RESP 端口 | Raft 端口 | 数据目录 |
|------|----|----------|----------|---------|
| Node 1 | `192.168.1.10` | `6379` | `16379` | `/data/lightkv/node1` |
| Node 2 | `192.168.1.11` | `6379` | `16379` | `/data/lightkv/node2` |
| Node 3 | `192.168.1.12` | `6379` | `16379` | `/data/lightkv/node3` |

### 3.2 启动命令

**参数说明**：

| 参数 | 说明 | 示例 |
|------|------|------|
| `--db-path` | 数据存储目录 | `/data/lightkv/node1` |
| `--tcp-port` | RESP 服务端口 | `6379` |
| `--http-port` | HTTP 监控端口 | `8080`（`0`=关闭） |
| `--cluster` | 启用 Raft 集群模式 | |
| `--node-id` | 本节点 ID（1-255，集群唯一） | `1` |
| `--raft-host` | Raft 内部通信监听地址 | `0.0.0.0` |
| `--raft-port` | Raft 内部通信端口 | `16379` |
| `--raft-peers` | 集群节点列表（见下方格式） | |

**raft-peers 格式**：
```
id1:host1:raft_port1:is_voter,id2:host2:raft_port2:is_voter,...
```

- `id`：节点 ID（与 `--node-id` 一致）
- `host`：节点 IP 或主机名
- `raft_port`：Raft 内部通信端口
- `is_voter`：`1` = 投票节点, `0` = 非投票节点

#### Node 1

```bash
mkdir -p /data/lightkv/node1
./lightkv_server \
  --db-path /data/lightkv/node1 \
  --tcp-port 6379 \
  --http-port 0 \
  --cluster \
  --node-id 1 \
  --raft-host 0.0.0.0 \
  --raft-port 16379 \
  --raft-peers "1:192.168.1.10:16379:1,2:192.168.1.11:16379:1,3:192.168.1.12:16379:1"
```

#### Node 2

```bash
mkdir -p /data/lightkv/node2
./lightkv_server \
  --db-path /data/lightkv/node2 \
  --tcp-port 6379 \
  --http-port 0 \
  --cluster \
  --node-id 2 \
  --raft-host 0.0.0.0 \
  --raft-port 16379 \
  --raft-peers "1:192.168.1.10:16379:1,2:192.168.1.11:16379:1,3:192.168.1.12:16379:1"
```

#### Node 3

```bash
mkdir -p /data/lightkv/node3
./lightkv_server \
  --db-path /data/lightkv/node3 \
  --tcp-port 6379 \
  --http-port 0 \
  --cluster \
  --node-id 3 \
  --raft-host 0.0.0.0 \
  --raft-port 16379 \
  --raft-peers "1:192.168.1.10:16379:1,2:192.168.1.11:16379:1,3:192.168.1.12:16379:1"
```

### 3.3 启动顺序

建议按 Node 1 → Node 2 → Node 3 顺序启动，间隔 2-3 秒：
- Node 1 先启动后会进入 Candidate 状态并发起选举（因只有自己，无法达成多数）
- Node 2、Node 3 启动后，集群自动完成选举，产生 Leader
- 首次启动约 3-5 秒内完成 Leader 选举

### 3.4 验证启动日志

**正常启动日志**：
```

[LightKV] Starting in Raft cluster mode (node_id=1)...
[LightKV] Raft engine initialized
[LightKV] Raft cluster started on 0.0.0.0:16379
[LightKV] Cluster initialized with 3 nodes

[LightKV] TCP listening on 0.0.0.0:6379
[Raft] Node 1 became FOLLOWER, term=1

```

```

```

## 4. 集群功能验证

### 4.1 基本连接

```bash
# 使用 nc 发送 RESP 命令
echo -e '*1\r\n$4\r\nPING\r\n' | nc -w 2 192.168.1.10 6379
# 预期输出: +PONG
```

### 4.2 CLUSTER INFO

```bash
echo -e '*2\r\n$7\r\nCLUSTER\r\n$4\r\nINFO\r\n' | nc -w 2 192.168.1.10 6379
```

**预期输出**：
```
cluster_state:ok
cluster_slots_assigned:16384
cluster_slots_ok:16384
cluster_known_nodes:3       ← 3 节点正常
cluster_size:3
cluster_current_epoch:...
cluster_node_id:1
cluster_slots_local:5461
```

### 4.3 CLUSTER KEYSLOT

```bash
echo -e '*3\r\n$7\r\nCLUSTER\r\n$7\r\nKEYSLOT\r\n$3\r\nKEY\r\n' | nc -w 2 192.168.1.10 6379
# 预期输出: :5305  （CRC16("KEY") & 16383）
```

### 4.4 CLUSTER RAFT-STATS

```bash
echo -e '*2\r\n$7\r\nCLUSTER\r\n$10\r\nRAFT-STATS\r\n' | nc -w 2 192.168.1.10 6379
```

**预期输出示例**：
```
term:45
commit_index:24
last_applied:24
log_count:25
role:follower          ← 本节点角色
leader_id:3            ← 当前 Leader 节点 ID
```

### 4.5 数据读写验证

```bash
# 1. 找到 Leader 节点
# 在任意节点执行 RAFT-STATS，leader_id 指示 Leader

# 2. 向 Leader 写入数据
echo -e '*3\r\n$3\r\nSET\r\n$3\r\nkey\r\n$5\r\nhello\r\n' | nc -w 2 192.168.1.11 6379
# 预期: +OK

# 3. 从 Leader 读取
echo -e '*2\r\n$3\r\nGET\r\n$3\r\nkey\r\n' | nc -w 2 192.168.1.11 6379
# 预期: $5\r\nhello

# 4. 从 Follower 读取（验证 Raft 复制）
echo -e '*2\r\n$3\r\nGET\r\n$3\r\nkey\r\n' | nc -w 2 192.168.1.10 6379
# 预期: $5\r\nhello  （数据已复制到所有节点）

echo -e '*2\r\n$3\r\nGET\r\n$3\r\nkey\r\n' | nc -w 2 192.168.1.12 6379
# 预期: $5\r\nhello
```

> **注意**：写入操作必须在 Leader 节点执行。写入 Follower 会返回 `-ERR raft propose failed`。客户端应通过 RAFT-STATS 定位 Leader，或由 SDK 自动转发。

### 4.6 使用 redis-cli

如果系统安装了 `redis-cli`，可直接使用标准 Redis 命令：

```bash
# 连接到任意节点
redis-cli -h 192.168.1.10 -p 6379

# 集群命令
127.0.0.1:6379> CLUSTER INFO
127.0.0.1:6379> CLUSTER KEYSLOT mykey
127.0.0.1:6379> CLUSTER RAFT-STATS

# 数据命令（需在 Leader 执行写操作）
127.0.0.1:6379> SET key hello
127.0.0.1:6379> GET key
```

## 5. 故障切换测试

### 5.1 测试场景：Kill Leader

```bash
# 1. 确认当前 Leader
echo -e '*2\r\n$7\r\nCLUSTER\r\n$10\r\nRAFT-STATS\r\n' | nc -w 2 192.168.1.10 6379
# 记录 leader_id

# 2. 在 Leader 节点上写入测试数据（通过 Leader 的 RESP 端口）
echo -e '*3\r\n$3\r\nSET\r\n$3\r\ntest\r\n$4\r\ndata\r\n' | nc -w 2 <LEADER_IP> 6379

# 3. 杀掉 Leader 进程
ssh user@<LEADER_IP> "kill \$(ps aux | grep '[l]ightkv_server.*node-id <LEADER_ID>' | awk '{print \$2}')"
# 或使用 pkill
ssh user@<LEADER_IP> "pkill -f 'lightkv_server.*node-id <LEADER_ID>'"

# 4. 等待选举（约 300-500ms）
sleep 2

# 5. 在任意存活节点上检查新 Leader
echo -e '*2\r\n$7\r\nCLUSTER\r\n$10\r\nRAFT-STATS\r\n' | nc -w 2 <SURVIVING_NODE_IP> 6379
# leader_id 应变为另一个节点

# 6. 验证数据仍然可读
echo -e '*2\r\n$3\r\nGET\r\n$3\r\ntest\r\n' | nc -w 2 <SURVIVING_NODE_IP> 6379
# 应返回 "data"
```

### 5.2 预期行为

| 事件 | 预期行为 | 耗时 |
|------|---------|------|
| Leader 进程被杀 | 所有节点检测到 Leader 失联 | < 150ms |
| Follower 发起选举 | 角色变为 Candidate，发起 RequestVote RPC | 150-300ms |
| 新 Leader 当选 | 获得多数派投票，提交 NoOp 日志 | < 500ms |
| 数据读写恢复 | 新 Leader 继续提供服务 | < 1s 总恢复时间 |

### 5.3 恢复被杀节点

```bash
# 重新启动被杀节点
ssh user@<KILLED_NODE_IP>
cd ~/lightkv/build
./lightkv_server \
  --db-path /data/lightkv/nodeX \
  --tcp-port 6379 \
  --http-port 0 \
  --cluster \
  --node-id <NODE_ID> \
  --raft-host 0.0.0.0 \
  --raft-port 16379 \
  --raft-peers "1:192.168.1.10:16379:1,2:192.168.1.11:16379:1,3:192.168.1.12:16379:1"

# 节点启动后自动从 Leader 同步缺失的日志
# 观察日志应有：Node X became FOLLOWER
```

## 7. 性能测试

### 7.1 内置基准测试

```bash
cd build && ./tests/benchmark_test
```

输出示例：
```
--- Phase 1: Write Benchmarks ---
SeqWrite(k8v100):      247K QPS
RandomWrite(k8v100):   182K QPS

--- Phase 2: Read Benchmarks ---
SeqRead(k8):           431K QPS
RandomRead(k8):        296K QPS
```

### 7.2 三节点集群吞吐

| 操作 | 单节点 QPS | 三节点 QPS | 延迟 P99 |
|------|-----------|-----------|---------|
| SET (1KB) | ~250K | 取决于 Leader 性能 | < 1ms |
| GET (1KB) | ~400K | ~1.2M（读扩散） | < 0.5ms |
| 批量写入 | ~200K | 同 Leader | < 2ms |

> 三节点集群的写吞吐受 Leader 单节点限制（Raft 复制需要 Leader 处理所有写入）。
> 读吞吐可通过 Follower 水平扩展（所有节点都可处理读请求）。

## 8. 常见问题排查

### 8.1 节点无法启动

**现象**：`bind: Address already in use`

**解决**：
```bash
# 查找占用端口的进程
ss -tlnp | grep -E '6379|16379'
# 杀掉旧进程再重试
kill <PID>
```

### 8.2 "-ERR raft propose failed"

**原因**：写操作发送到了 Follower 节点，而非 Leader。

**解决**：
- 通过 `CLUSTER RAFT-STATS` 查询 Leader
- 将写操作重定向到 Leader
- 后续可通过 SDK 自动转发（`ForwardPropose` 机制）

### 8.3 节点启动后状态异常

**现象**：日志显示重复选举（term 持续增长），无法选出 Leader

**排查步骤**：
```bash
# 1. 检查所有节点是否已启动
ps aux | grep lightkv_server

# 2. 检查 Raft 端口是否互通
echo -e '*1\r\n$4\r\nPING\r\n' | nc -w 2 <OTHER_NODE_IP> 16379  # 不应有响应（二进制协议）

# 3. 检查 RESP 端口
echo -e '*1\r\n$4\r\nPING\r\n' | nc -w 2 <OTHER_NODE_IP> 6379
# 预期: +PONG

# 4. 检查节点 ID 是否冲突
# --raft-peers 中的节点 ID 必须唯一
```

### 8.4 数据无法跨节点读取

**现象**：写入 Leader 成功后，Follower 读取不到数据

**排查**：
```bash
# 1. 检查 RAFT-STATS 中的 commit_index（应在持续增长）
CLUSTER RAFT-STATS

# 2. 检查节点日志是否有复制失败信息
grep -i "replicate\|AppendEntries\|snapshot" /data/lightkv/node*/lightkv_server.log

# 3. 确认 Follower 节点 RESP 端口可达
redis-cli -h <FOLLOWER_IP> -p 6379 PING
```

### 8.5 "非法指令 (核心已转储)"

**原因**：二进制在较新的 CPU 上编译，运行在旧 CPU 上。

**解决**：
```bash
# 在所有节点上重新编译（或统一编译环境）
cmake .. -DCMAKE_BUILD_TYPE=Release -DLIGHTKV_BUILD_TESTS=OFF
make -j$(nproc) lightkv_server
```

### 8.6 调试日志

所有节点启动后，Raft 日志输出在 stderr，可通过重定向查看：

```bash
# 通过 nohup 日志查看
tail -f /tmp/lkv_e2e1.log | grep "\[Raft\]"

# 重点关注
[Raft] Node X became CANDIDATE    → 选举开始
[Raft] Node X became LEADER       → 新 Leader 当选
[Raft] Node X became FOLLOWER     → 收到更高 term，退为 Follower
[Raft] Starting election term=Y   → 发起选举
[Raft] Sending snapshot to follower X  → 快照传输
```

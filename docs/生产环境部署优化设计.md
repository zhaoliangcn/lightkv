### 2.1 完整 Compaction 实现

> **状态**: ✅ 已完成（见 2.0.1）

#### 2.1.1 背景与问题

当前 `CompactionWorker::DoCompaction()` 为空实现（[compaction.cpp](file:///Users/macmima1234/code/mykvdb/lightkv/src/compaction.cpp#L22-L27)）：

```cpp
void CompactionWorker::DoCompaction(Compaction* compaction) {
    if (compaction->inputs_[0].empty()) return;
    // 空实现，仅注释
}
```

这意味着：
- L0 SSTable 文件**无限增长**
- 读性能随 L0 文件数**线性恶化**
- 无法运行删除标记的物理清除
- 磁盘空间**只增不减**
- 重启后 L0 中存在大量重叠文件，读取需遍历所有文件

#### 2.1.2 设计方案

##### Leveled Compaction 策略

采用 Leveled Compaction（与 LevelDB/RocksDB 相同策略），核心思想：每一层的数据总大小按 `level_multiplier`（默认 10）指数增长，确保新写入的数据尽快合并。

```
目标大小公式:
  Level 0:  触发阈值 = l0_file_num_trigger (默认 4 个文件)
  Level i:  目标大小 = base_level_size * (level_multiplier)^i  (i ≥ 1)
  
  其中 base_level_size = options_.memtable_size * level_multiplier
```

##### Compaction 调度算法

```cpp
CompactionScore DBImpl::PickCompaction() {
    // 1. 检查 L0：文件数是否超过阈值
    if (levels_[0].size() >= options_.l0_file_num_trigger) {
        return {0, static_cast<double>(levels_[0].size()) / options_.l0_file_num_trigger};
    }
    
    // 2. 检查 L1-L6：总大小是否超过层目标
    uint64_t base_size = options_.memtable_size * options_.level_multiplier;
    for (int i = 1; i < options_.max_level; ++i) {
        uint64_t level_size = 0;
        for (const auto& file : levels_[i]) {
            level_size += file->FileSize();
        }
        uint64_t target_size = base_size * static_cast<uint64_t>(
            std::pow(options_.level_multiplier, i));
        
        if (level_size > target_size) {
            return {i, static_cast<double>(level_size) / target_size};
        }
    }
    
    return {-1, 0};  // 无需 compaction
}
```

##### 多路归并合并算法

```cpp
// 核心合并逻辑
class MergingIterator {
public:
    // 从多个 SSTable 中构造合并迭代器
    // 按 key 字典序升序，相同 key 取最大 seq（最新版本）
    
    bool Valid() const;
    void Next();
    Slice key() const;
    Slice value() const;
    uint64_t seq() const;
    bool IsDeleted() const;
};

void CompactionWorker::DoCompaction(Compaction* compaction) {
    // Step 1: 收集所有输入文件
    std::vector<std::shared_ptr<SSTable>> all_inputs;
    all_inputs.insert(all_inputs.end(), 
                      compaction->inputs_[0].begin(), compaction->inputs_[0].end());
    all_inputs.insert(all_inputs.end(), 
                      compaction->inputs_[1].begin(), compaction->inputs_[1].end());
    
    if (all_inputs.empty()) return;
    
    // Step 2: 创建 TableBuilder 用于输出
    TableBuilder builder(options_, output_filename);
    
    // Step 3: 多路归并
    MergingIterator merge_iter(all_inputs);
    merge_iter.SeekToFirst();
    
    std::string last_key;
    uint64_t output_file_size = 0;
    
    while (merge_iter.Valid()) {
        // 跳过相同 key 的旧版本
        if (!last_key.empty() && merge_iter.key() == Slice(last_key)) {
            merge_iter.Next();
            continue;
        }
        
        // 跳过删除标记（如果删除标记不是最新版本则不处理）
        if (!merge_iter.IsDeleted()) {
            builder.Add(merge_iter.key(), merge_iter.value());
            last_key = merge_iter.key().ToString();
        }
        
        merge_iter.Next();
        
        // 输出文件超过目标大小 → 切换到新文件
        // 目标文件大小 ≈ 层目标大小 / 文件数期望值
        if (builder.FileSize() > target_file_size) {
            builder.Finish();
            // 打开新文件并加入 VersionEdit
            builder = TableBuilder(options_, next_output_filename);
        }
    }
    
    builder.Finish();
    
    // Step 4: 更新元数据
    // 删除旧文件，添加新文件
    for (auto& file : all_inputs) {
        std::remove(file->filename().c_str());
    }
}
```

##### 文件选择策略

L0 Compaction：选取所有 L0 文件 + L1 中与这些文件 key 范围重叠的文件

```
L0 文件 key 范围: [aaa, ccc], [bbb, ddd], [eee, fff]
L1 文件 key 范围: [aaa, bbb], [ccc, ddd], [eee, ggg]
选中范围: [aaa, fff] → L1 中有 2 个文件重叠 → 选取 L0 全部 3 个 + L1 的 2 个
```

Ln (n≥1) Compaction：选取该层中 **最小 key 的文件** + 下层重叠文件

```
L1 选中文件 key: [aaa, ccc]
L2 重叠文件 key: [aaa, bbb], [bbb, ddd] → 选中这 2 个
```

```cpp
    data_block_index_ = 0;
    SwitchToBlock(0);  // 打开第一个 Data Block

void SSTable::Iterator::Next() {
    if (!Valid()) return;
    data_iter_->Next();
    if (!data_iter_->Valid()) {
        // 当前块遍历完毕 → 切换到下一个块
        ++data_block_index_;
        if (data_block_index_ < table_->data_block_handles_.size()) {
            SwitchToBlock(data_block_index_);
            data_iter_->SeekToFirst();
        }
    }
}

void SSTable::Iterator::SwitchToBlock(int index) {
    BlockContents contents;
    auto& handle = table_->data_block_handles_[index];
    table_->ReadBlock(handle, &contents);
    data_iter_ = std::make_unique<Block::Iterator>(
        new Block(contents.data.data(), contents.data.size()));
}
```

```cpp
void DBImpl::Iterator::SeekToFirst() {
    // Step 1: 收集所有数据源的迭代器
    std::vector<IteratorSource> sources;
    
    // MemTable
    sources.push_back({SourceType::kMemTable, mem_iter, ...});
    // Immutable MemTable
    if (imm_) sources.push_back({SourceType::kImmTable, imm_iter, ...});
    // 每一层
    for (int level = 0; level < 7; ++level) {
        for (auto& table : levels_[level]) {
            sources.push_back({SourceType::kSSTable, table->NewIterator(), ...});
        }
    }
    
    // Step 2: 合并所有迭代器
    merge_iter_.Reset(sources);
    merge_iter_.SeekToFirst();
    
    // Step 3: 读取第一个有效值
    UpdateCurrent();
}

void DBImpl::Iterator::Seek(const Slice& target) {
    merge_iter_.Seek(target);
    UpdateCurrent();
}

void DBImpl::Iterator::Next() {
    merge_iter_.Next();
    UpdateCurrent();
}

void DBImpl::Iterator::UpdateCurrent() {
    // 跳过删除标记：如果当前 key 的最新版本是删除标记，
    // 则继续 Next() 直到找到有效值或结束
    while (merge_iter_.Valid()) {
        const auto& entry = merge_iter_.Current();
        if (!entry.is_deleted) {
            valid_ = true;
            key_ = entry.key.ToString();
            value_ = entry.value.ToString();
            return;
        }
        merge_iter_.Next();
    }
    valid_ = false;
}
```

### 2.3 数据完整性校验

> **状态**: ✅ 已完成（见 2.0.3）

#### 2.3.1 背景与问题

[options.h](file:///Users/macmima1234/code/mykvdb/lightkv/include/lightkv/options.h#L23) 中 `paranoid_checks` 选项存在但从未被任何代码读取使用。

现有校验基础：
- WAL 每条记录有 **CRC32 校验**（[wal.cpp](file:///Users/macmima1234/code/mykvdb/lightkv/src/wal.cpp#L62-L69)）

这意味着静默数据损坏无法被检测。

#### 2.3.2 设计方案

##### SSTable Data Block CRC32

在 SSTable 的每个 Data Block 末尾追加 CRC32 校验和。

```
当前 Block 编码:
┌────────────────┬──────────────────┬────┬──────────┐
│ KV Entries     │ Restart Offsets  │ #  │ Num      │
│ (prefix compr) │ (uint32 array)   │    │ Restarts │
└────────────────┴──────────────────┴────┴──────────┘

优化后 Block 编码:
```

> **状态**: ✅ 已完成（见 2.0.4）

#### 2.4.1 当前问题

1. **读写互斥**：`Get()` 和 `Put()` 使用同一个 `mutex_`，导致读操作阻塞写操作
2. **嵌套锁**：`Get()` 获取 `mutex_` 后，`SearchSSTable()` 再次获取 `mutex_` → 虽因可重入未死锁，但设计不当
3. **无锁路径**：后台线程与前台线程共享 `levels_` 向量，可能导致**迭代器失效**


##### 读写锁（std::shared_mutex）

```cpp

    mutable std::shared_mutex rw_mutex_;  // 替换 std::mutex mutex_
```

##### 锁层级定义

| 锁 | 保护数据 | 加锁方式 |
|----|---------|---------|
| `rw_mutex_` | `mem_`, `imm_`, `levels_[7]`, `has_imm_`, `bg_scheduled_` | 读：`shared_lock`，写：`unique_lock` |
| `bg_mutex_` | `next_file_id_`, `wal_` 切换 | `unique_lock`（仅在后台线程用） |

##### 接口加锁规则

```cpp

    mem_->Insert(seq, key, value);
    // ...

    std::shared_lock lock(rw_mutex_);
    // 同时允许多个 Get 进入
    mem_->Get(key, value, snapshot);
    // ...
}

// FlushMemTable - MemTable 切换 → 独占锁

    std::unique_lock lock(rw_mutex_);
    imm_ = mem_;
    // ...
}

// WriteLevel0Table - 添加新 SSTable → 独占锁

    // TableBuilder 在锁外执行
    std::unique_lock lock(rw_mutex_);
    levels_[0].push_back(table);
}

// BackgroundCompaction - 修改 levels_ → 独占锁
void BackgroundCompaction() {
    std::unique_lock lock(rw_mutex_);
    levels_[0].clear();
```

##### SSTable 生命周期安全

当前问题：后台线程在 Compaction 中删除 SSTable 时，前台 `Get()` 可能正在读取。

解决方案：使用 **引用计数 + 延迟删除**。

```cpp
// 方案 A: shared_ptr 延迟释放
// 当后台线程从 levels_ 中移除 SSTable 时，
// 只是从 vector 中移除，shared_ptr 本身由还在读的线程持有

// 后台线程：
{
    std::unique_lock lock(rw_mutex_);
    // 移除：shared_ptr 引用计数 -1
    // 前台线程可能仍持有 shared_ptr 副本 → 正常读取
    levels_[0].erase(levels_[0].begin() + i);
}
// 所有前台线程完成读取后，shared_ptr 自动释放

// 方案 B: 墓碑标记
// 如果方案 A 不能满足需求（如需要立即关闭文件描述符），可以使用墓碑标记
struct SSTableEntry {
    std::shared_ptr<SSTable> table;
    bool deleted = false;
};
```

方案 A 简单可靠，推荐优先采用。

```cpp
DBImpl::~DBImpl() {
    // Step 1: 触发最终 Flush
    {
        std::unique_lock<std::mutex> lock(mutex_);
        if (!mem_->empty()) {
            // 将当前 MemTable 标记为不可变，等待 Flush
            imm_ = mem_;
            has_imm_ = true;
            mem_ = std::make_shared<MemTable>();
        }
    }
    
    // Step 2: 等待所有 Flush/Compaction 完成
    {
        std::unique_lock<std::mutex> lock(mutex_);
        bg_cv_.wait(lock, [this] {
            return !has_imm_ && !bg_scheduled_;
        });
    }
    
    // Step 3: 通知后台线程退出
    shutting_down_ = true;
    bg_cv_.notify_all();
    if (bg_thread_.joinable()) bg_thread_.join();
    
    // Step 4: 关闭 WAL（截断到实际写入位置）
    if (wal_) {
        wal_->Sync();
        wal_->Close();
    }
    
    // Step 5: 可选 - 同步所有 SSTable 文件
    SyncAllSSTables();
}
```

```cpp
        enum Type { kPut, kDelete };
        Type type;
        std::string key;
        std::string value;
    };
    std::vector<WriteOp> write_buf_;
};
```

#### 4.1.2 实现要点

- **原子提交**：`Commit()` 时在一个 WAL 记录中标记事务边界

- **冲突检测**：在 `Commit()` 时检查 key 是否被其他事务修改
- **读己之写**：事务内 `Get()` 优先读取本地写缓冲

```cpp
        // 根据类型选择压缩算法

}
```

**关键发现**:
- **单线程性能**: LightKV 在常规操作下比 Redis 快 **4.8x~4.9x**，Pipeline 模式下快 **1.2x~1.9x**
- **并发性能**: LightKV 在 50/100 并发下吞吐量是 Redis 的 **4.4x~4.8x**
- **延迟优势**: LightKV P50 延迟是 Redis 的 **1/4.7**，P99 延迟是 Redis 的 **1/2.4~1/4.3**
- **高并发稳定性**: LightKV 在 100 并发下吞吐量保持稳定（42k ops/s），而 Redis 从 9.7k 降至 8.9k ops/s
- **原因分析**: LightKV 基于 LSM-Tree 的写入优化设计（WAL + MemTable）在高并发写入场景下具有天然优势；Redis 单线程事件循环模型在并发客户端增多时成为瓶颈

### 7.2 阻塞生产部署的问题

#### 🔴 P0-1：无认证机制（高危）

| 项目 | 说明 |
|------|------|
| **问题** | ~~没有 `AUTH` 命令，任何人连接 TCP 端口即可读写全部数据~~ |
| **状态** | ✅ **已解决**（v1.9） |
| **实现** | 添加 `CONFIG SET requirepass` + `AUTH` 命令，支持简单密码认证 |
| **代码变更** | `server.h` 新增 `requirepass` 配置项；`server.cpp` 实现 `handle_auth` 和 `handle_config`；`client.h/cpp` 新增 `Auth()`/`ConfigGet()`/`ConfigSet()` 方法 |
| **测试** | `tests/test_auth.cpp` 16 个测试用例全部通过 |
| **后续** | 可扩展 TLS 加密传输、ACL 访问控制 |

#### 🔴 P0-2：单线程事件循环（架构瓶颈）

| 项目 | 说明 |
|------|------|
| **问题** | TCP Server 使用单线程 kqueue/epoll 事件循环，所有命令在事件循环中同步执行 |
| **影响** | 一个慢命令（如 `KEYS *`）会阻塞所有连接；无法利用多核 CPU |
| **代码位置** | [server.cpp:200-230](file:///Users/macmima1234/code/mykvdb/lightkv/src/server.cpp#L200-L230) — 单线程事件循环 |
| **建议方案** | 引入 Worker Pool（线程池），事件循环仅负责 I/O 收发，命令执行派发至 Worker 线程 |

#### 🔴 P0-3：TTL 仅惰性删除（磁盘膨胀）

| 项目 | 说明 |
|------|------|
| **问题** | 没有后台过期扫描线程，过期 key 只在被访问时删除 |
| **影响** | 大量设置 TTL 的 key 过期后持续占用磁盘空间，直到 Compaction 清理；读取过期 key 产生不必要 I/O |
| **建议方案** | 添加后台过期扫描线程（类似 Redis 的 active expire），定期抽样扫描并删除过期 key |

#### 🔴 P0-4：无优雅信号处理

| 项目 | 说明 |
|------|------|
| **问题** | 未注册 SIGINT/SIGTERM 信号处理，`Server::Stop()` 需要外部调用 |
| **影响** | Ctrl+C 直接退出可能导致 WAL 未 flush、MemTable 未持久化，存在丢数据风险 |
| **代码证据** | 代码中无 `signal()` / `sigaction()` 调用 |
| **建议方案** | 注册 SIGINT/SIGTERM 信号处理，捕获后调用 `Server::Stop()` → `DB::Close()` 确保数据落盘 |

### 7.3 重要但非阻塞的问题

#### 🟡 P1-1：缺少关键运维命令

| 缺失命令 | 影响 | 建议优先级 |

| ~~`AUTH`~~ | ~~无法认证~~ | ~~P0~~ ✅ 已实现 |

#### 🟡 P1-2：连接管理不完善

| 项目 | 说明 |
|------|------|
| **连接数限制** | `max_connections=1024` 硬编码，无动态调整 |
| **空闲超时** | 无 idle timeout 踢除机制，僵尸连接占用资源 |
| **慢客户端防护** | recv buf 无上限，慢客户端可能导致内存无限增长 |
| **建议方案** | 添加 `timeout` 配置（空闲 N 秒断开）、recv buf 上限（如 64MB）、连接数动态告警 |

#### 🟡 P1-3：无日志系统

| 项目 | 说明 |

| **影响** | 生产问题排查困难，无法追溯历史事件 |
| **建议方案** | 引入轻量级日志库（如 spdlog）或自建日志模块，支持级别过滤和文件轮转 |

#### 🟡 P1-4：无复制/高可用

| 项目 | 说明 |
|------|------|
| **当前状态** | 单实例运行，无主从复制、无哨兵、无集群 |
| **影响** | 单点故障，数据仅存一台机器，机器故障即数据不可用 |
| **建议方案** | 长期规划：实现基于 WAL 重放的主从复制，配合哨兵实现自动故障转移 |

### 7.6 结论

**LightKV 目前适合用于本地开发、测试环境和内网非关键服务，不建议直接部署到生产核心环境。**

核心优势：
- 存储引擎性能优秀（比 Redis 快 4.8x）
- 数据类型覆盖全面（8 种，83 条命令）
- 测试充分（256 个用例全部通过）
- 多语言 SDK 支持（C++/Node.js/Python/Go）

主要短板：
- **安全性**：无认证、无 TLS
- **运维能力**：无日志、无信号处理、缺少运维命令
- **高可用**：无复制、单点故障

完成 Step 1-2（约 2-4 周）后可达到内网生产可用标准。

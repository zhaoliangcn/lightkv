# LightKV 生产环境部署优化设计文档

## 1. 文档概述

### 1.1 目标

本文档旨在定义 LightKV v1.0 达到生产部署条件所需完成的技术优化方案。通过对现有代码的全面审计，识别出 **P0（阻塞级）**、**P1（重要级）** 和 **P2（增强级）** 三个优先级的问题，并为每个问题提供完整的设计方案。

### 1.2 当前状态摘要

| 维度 | 状态 | 说明 |
|------|------|------|
| 架构设计 | ✅ 合理 | LSM-Tree 7 层，WAL + MemTable + SSTable |
| 写入性能 | ✅ 优秀 | 小值 908K QPS，P99 3.2μs |
| 读取性能 | ✅ 优秀 | 顺序读 1.9M QPS，P99 1.1μs |

| 数据校验 | ❌ 未启用 | paranoid_checks 选项存在但未使用 |
| 线程安全 | ⚠️ 有风险 | 互斥锁嵌套、无读写锁 |
| 优雅关闭 | ❌ 缺失 | 析构时不 Flush 数据 |
| 监控指标 | ❌ 缺失 | 无法观测运行时状态 |

## 2. P0 — 阻塞级优化（必须完成）

### 2.1 完整 Compaction 实现

#### 2.1.1 背景与问题

当前 `CompactionWorker::DoCompaction()` 为空实现（[compaction.cpp](file:///Users/macmima1234/code/mykvdb/lightkv/src/compaction.cpp#L22-L27)）：

```cpp
void CompactionWorker::DoCompaction(Compaction* compaction) {
    if (compaction->inputs_[0].empty()) return;
    // 空实现，仅注释
}
```

这意味着：
- L0 SSTable 文件**无限增长**
- 读性能随 L0 文件数**线性恶化**
- 无法运行删除标记的物理清除
- 磁盘空间**只增不减**
- 重启后 L0 中存在大量重叠文件，读取需遍历所有文件

#### 2.1.2 设计方案

##### Leveled Compaction 策略

采用 Leveled Compaction（与 LevelDB/RocksDB 相同策略），核心思想：每一层的数据总大小按 `level_multiplier`（默认 10）指数增长，确保新写入的数据尽快合并。

```
目标大小公式:
  Level 0:  触发阈值 = l0_file_num_trigger (默认 4 个文件)
  Level i:  目标大小 = base_level_size * (level_multiplier)^i  (i ≥ 1)
  
  其中 base_level_size = options_.memtable_size * level_multiplier
```

##### Compaction 调度算法

```cpp
CompactionScore DBImpl::PickCompaction() {
    // 1. 检查 L0：文件数是否超过阈值
    if (levels_[0].size() >= options_.l0_file_num_trigger) {
        return {0, static_cast<double>(levels_[0].size()) / options_.l0_file_num_trigger};
    }
    
    // 2. 检查 L1-L6：总大小是否超过层目标
    uint64_t base_size = options_.memtable_size * options_.level_multiplier;
    for (int i = 1; i < options_.max_level; ++i) {
        uint64_t level_size = 0;
        for (const auto& file : levels_[i]) {
            level_size += file->FileSize();
        }
        uint64_t target_size = base_size * static_cast<uint64_t>(
            std::pow(options_.level_multiplier, i));
        
        if (level_size > target_size) {
            return {i, static_cast<double>(level_size) / target_size};
        }
    }
    
    return {-1, 0};  // 无需 compaction
}
```

##### 多路归并合并算法

```cpp
// 核心合并逻辑
class MergingIterator {
public:
    // 从多个 SSTable 中构造合并迭代器
    // 按 key 字典序升序，相同 key 取最大 seq（最新版本）
    
    bool Valid() const;
    void Next();
    Slice key() const;
    Slice value() const;
    uint64_t seq() const;
    bool IsDeleted() const;
};

void CompactionWorker::DoCompaction(Compaction* compaction) {
    // Step 1: 收集所有输入文件
    std::vector<std::shared_ptr<SSTable>> all_inputs;
    all_inputs.insert(all_inputs.end(), 
                      compaction->inputs_[0].begin(), compaction->inputs_[0].end());
    all_inputs.insert(all_inputs.end(), 
                      compaction->inputs_[1].begin(), compaction->inputs_[1].end());
    
    if (all_inputs.empty()) return;
    
    // Step 2: 创建 TableBuilder 用于输出
    TableBuilder builder(options_, output_filename);
    
    // Step 3: 多路归并
    MergingIterator merge_iter(all_inputs);
    merge_iter.SeekToFirst();
    
    std::string last_key;
    uint64_t output_file_size = 0;
    
    while (merge_iter.Valid()) {
        // 跳过相同 key 的旧版本
        if (!last_key.empty() && merge_iter.key() == Slice(last_key)) {
            merge_iter.Next();
            continue;
        }
        
        // 跳过删除标记（如果删除标记不是最新版本则不处理）
        if (!merge_iter.IsDeleted()) {
            builder.Add(merge_iter.key(), merge_iter.value());
            last_key = merge_iter.key().ToString();
        }
        
        merge_iter.Next();
        
        // 输出文件超过目标大小 → 切换到新文件
        // 目标文件大小 ≈ 层目标大小 / 文件数期望值
        if (builder.FileSize() > target_file_size) {
            builder.Finish();
            // 打开新文件并加入 VersionEdit
            builder = TableBuilder(options_, next_output_filename);
        }
    }
    
    builder.Finish();
    
    // Step 4: 更新元数据
    // 删除旧文件，添加新文件
    for (auto& file : all_inputs) {
        std::remove(file->filename().c_str());
    }
}
```

##### 文件选择策略

L0 Compaction：选取所有 L0 文件 + L1 中与这些文件 key 范围重叠的文件

```
L0 文件 key 范围: [aaa, ccc], [bbb, ddd], [eee, fff]
L1 文件 key 范围: [aaa, bbb], [ccc, ddd], [eee, ggg]
选中范围: [aaa, fff] → L1 中有 2 个文件重叠 → 选取 L0 全部 3 个 + L1 的 2 个
```

Ln (n≥1) Compaction：选取该层中 **最小 key 的文件** + 下层重叠文件

```
L1 选中文件 key: [aaa, ccc]
L2 重叠文件 key: [aaa, bbb], [bbb, ddd] → 选中这 2 个
```

### 2.2 DB Iterator 实现

#### 2.2.1 背景与问题

当前 [iterator.cpp](file:///Users/macmima1234/code/mykvdb/lightkv/src/iterator.cpp) 仅为空文件（11 行），`DBImpl::Iterator` 类声明在 [db_impl.h](file:///Users/macmima1234/code/mykvdb/lightkv/include/lightkv/db_impl.h#L36-L48) 中但方法均未实现。

没有 Iterator 意味着：
- 无法进行范围查询（`Seek("user:1000") → Next() × 100`）
- 无法进行全表扫描
- 后端无法实现备份/导出功能

#### 2.2.2 设计方案

DB Iterator 需要**合并**三个数据源：
1. 活跃 MemTable
2. Immutable MemTable（如果有）
3. 所有 7 层的 SSTable

按 key 字典序合并，相同 key 取最新版本（最大 seq），跳过删除标记。

##### 层级合并迭代器架构

```
DBImpl::Iterator
      │
      ├── MemTable Iterator (SkipList)
      ├── Immutable MemTable Iterator (SkipList)
      └── Level Iterator (L0, L1, ..., L6)
              │
              ├── L0: 逐个 SSTable Iterator → 合并（key 重叠）
              └── L1-L6: 分段 SSTable Iterator → 依次（key 不重叠）
```

```cpp
void SSTable::Iterator::SeekToFirst() {
    data_block_index_ = 0;
    SwitchToBlock(0);  // 打开第一个 Data Block
    if (data_iter_) data_iter_->SeekToFirst();
}

void SSTable::Iterator::Next() {
    if (!Valid()) return;
    data_iter_->Next();
    if (!data_iter_->Valid()) {
        // 当前块遍历完毕 → 切换到下一个块
        ++data_block_index_;
        if (data_block_index_ < table_->data_block_handles_.size()) {
            SwitchToBlock(data_block_index_);
            data_iter_->SeekToFirst();
        }
    }
}

void SSTable::Iterator::SwitchToBlock(int index) {
    BlockContents contents;
    auto& handle = table_->data_block_handles_[index];
    table_->ReadBlock(handle, &contents);
    data_iter_ = std::make_unique<Block::Iterator>(
        new Block(contents.data.data(), contents.data.size()));
}
```

```cpp
void DBImpl::Iterator::SeekToFirst() {
    // Step 1: 收集所有数据源的迭代器
    std::vector<IteratorSource> sources;
    
    // MemTable
    sources.push_back({SourceType::kMemTable, mem_iter, ...});
    // Immutable MemTable
    if (imm_) sources.push_back({SourceType::kImmTable, imm_iter, ...});
    // 每一层
    for (int level = 0; level < 7; ++level) {
        for (auto& table : levels_[level]) {
            sources.push_back({SourceType::kSSTable, table->NewIterator(), ...});
        }
    }
    
    // Step 2: 合并所有迭代器
    merge_iter_.Reset(sources);
    merge_iter_.SeekToFirst();
    
    // Step 3: 读取第一个有效值
    UpdateCurrent();
}

void DBImpl::Iterator::Seek(const Slice& target) {
    merge_iter_.Seek(target);
    UpdateCurrent();
}

void DBImpl::Iterator::Next() {
    merge_iter_.Next();
    UpdateCurrent();
}

void DBImpl::Iterator::UpdateCurrent() {
    // 跳过删除标记：如果当前 key 的最新版本是删除标记，
    // 则继续 Next() 直到找到有效值或结束
    while (merge_iter_.Valid()) {
        const auto& entry = merge_iter_.Current();
        if (!entry.is_deleted) {
            valid_ = true;
            key_ = entry.key.ToString();
            value_ = entry.value.ToString();
            return;
        }
        merge_iter_.Next();
    }
    valid_ = false;
}
```

### 2.3 数据完整性校验

#### 2.3.1 背景与问题

[options.h](file:///Users/macmima1234/code/mykvdb/lightkv/include/lightkv/options.h#L23) 中 `paranoid_checks` 选项存在但从未被任何代码读取使用。

现有校验基础：
- WAL 每条记录有 **CRC32 校验**（[wal.cpp](file:///Users/macmima1234/code/mykvdb/lightkv/src/wal.cpp#L62-L69)）
- SSTable **没有块级校验**，只有 Footer 的 Magic Number（"LIGHTKV1"）

这意味着静默数据损坏无法被检测。

#### 2.3.2 设计方案

##### SSTable Data Block CRC32

在 SSTable 的每个 Data Block 末尾追加 CRC32 校验和。

```
当前 Block 编码:
┌────────────────┬──────────────────┬────┬──────────┐
│ KV Entries     │ Restart Offsets  │ #  │ Num      │
│ (prefix compr) │ (uint32 array)   │    │ Restarts │
└────────────────┴──────────────────┴────┴──────────┘

优化后 Block 编码:
┌────────────────┬──────────────────┬────┬──────────┬────────┐
│ KV Entries     │ Restart Offsets  │ #  │ Num      │ CRC32  │
│ (prefix compr) │ (uint32 array)   │    │ Restarts │ 4 bytes│
└────────────────┴──────────────────┴────┴──────────┴────────┘
```

修改 [block_builder.cpp](file:///Users/macmima1234/code/mykvdb/lightkv/src/block_builder.cpp)：

```cpp
Slice BlockBuilder::Finish() {
    // 原有逻辑：写入重启点数组
    for (uint32_t i = 0; i < num_restarts_; ++i) {
        PutFixed32(&buffer_, restarts_[i]);
    }
    PutFixed32(&buffer_, num_restarts_);

    finished_ = true;
    return Slice(buffer_);
}
```

### 2.4 线程安全重构

#### 2.4.1 当前问题

1. **读写互斥**：`Get()` 和 `Put()` 使用同一个 `mutex_`，导致读操作阻塞写操作
2. **嵌套锁**：`Get()` 获取 `mutex_` 后，`SearchSSTable()` 再次获取 `mutex_` → 虽因可重入未死锁，但设计不当
3. **无锁路径**：后台线程与前台线程共享 `levels_` 向量，可能导致**迭代器失效**

```cpp
// 问题 1: 读写不分离
Status DBImpl::Get(...) {
    std::lock_guard<std::mutex> lock(mutex_);  // 写操作也被阻塞
    
    // 问题 2: 嵌套加锁
    Status s = SearchSSTable(level, key, value, snapshot);
}

Status DBImpl::SearchSSTable(...) const {
    std::lock_guard<std::mutex> lock(mutex_);  // 嵌套锁
    for (const auto& table : files) {
        table->Get(key, value, &seq);  // table 是 shared_ptr，此时可能被后台线程删除
    }
}
```

#### 2.4.2 设计方案

##### 读写锁（std::shared_mutex）

```cpp
class DBImpl : public DB {
    mutable std::shared_mutex rw_mutex_;  // 替换 std::mutex mutex_
    
    // Flush/Compaction 专用锁（不阻塞前台读）
    std::mutex bg_mutex_;  
};
```

##### 锁层级定义

| 锁 | 保护数据 | 加锁方式 |
|----|---------|---------|
| `rw_mutex_` | `mem_`, `imm_`, `levels_[7]`, `has_imm_`, `bg_scheduled_` | 读：`shared_lock`，写：`unique_lock` |
| `bg_mutex_` | `next_file_id_`, `wal_` 切换 | `unique_lock`（仅在后台线程用） |

##### 接口加锁规则

```cpp
// Put/Delete - 写路径 → 独占锁
Status Put(...) {
    std::unique_lock lock(rw_mutex_);
    mem_->Insert(seq, key, value);
    // ...
}

// Get - 读路径 → 共享锁
Status Get(...) {
    std::shared_lock lock(rw_mutex_);
    // 同时允许多个 Get 进入
    mem_->Get(key, value, snapshot);
    // ...
}

// FlushMemTable - MemTable 切换 → 独占锁
void FlushMemTable() {
    std::unique_lock lock(rw_mutex_);
    imm_ = mem_;
    // ...
}

// WriteLevel0Table - 添加新 SSTable → 独占锁
Status WriteLevel0Table(...) {
    // TableBuilder 在锁外执行
    std::unique_lock lock(rw_mutex_);
    levels_[0].push_back(table);
}

// BackgroundCompaction - 修改 levels_ → 独占锁
void BackgroundCompaction() {
    std::unique_lock lock(rw_mutex_);
    levels_[0].clear();
}
```

##### SSTable 生命周期安全

当前问题：后台线程在 Compaction 中删除 SSTable 时，前台 `Get()` 可能正在读取。

解决方案：使用 **引用计数 + 延迟删除**。

```cpp
// 方案 A: shared_ptr 延迟释放
// 当后台线程从 levels_ 中移除 SSTable 时，
// 只是从 vector 中移除，shared_ptr 本身由还在读的线程持有

// 后台线程：
{
    std::unique_lock lock(rw_mutex_);
    // 移除：shared_ptr 引用计数 -1
    // 前台线程可能仍持有 shared_ptr 副本 → 正常读取
    levels_[0].erase(levels_[0].begin() + i);
}
// 所有前台线程完成读取后，shared_ptr 自动释放

// 方案 B: 墓碑标记
// 如果方案 A 不能满足需求（如需要立即关闭文件描述符），可以使用墓碑标记
struct SSTableEntry {
    std::shared_ptr<SSTable> table;
    bool deleted = false;
};
```

方案 A 简单可靠，推荐优先采用。

#### 2.4.3 性能预期

|------|------|--------|------|

| 读写混合 | 读写互相阻塞 | 读不阻塞写 | 显著 |

## 3. P1 — 重要级优化

### 3.1 优雅关闭与数据保护

#### 3.1.1 当前问题

[DBImpl 析构函数](file:///Users/macmima1234/code/mykvdb/lightkv/src/db_impl.cpp#L30-L41)：

```cpp
DBImpl::~DBImpl() {
    shutting_down_ = true;
    bg_cv_.notify_all();
    if (bg_thread_.joinable()) bg_thread_.join();
    if (wal_) wal_->Close();
    // 未 Flush 未完成的 MemTable！
    // 可能有活跃的 Immutable MemTable 未刷入 SSTable
}
```

这意味着：
- 进程退出时，MemTable 中尚未持久化的数据通过 WAL 保存（WAL 已关闭）
- 但 Immutable MemTable 如果正在 Flush，可能中途终止，数据丢失
- WAL 关闭后发生崩溃，数据不可恢复

#### 3.1.2 设计方案

```cpp
DBImpl::~DBImpl() {
    // Step 1: 触发最终 Flush
    {
        std::unique_lock<std::mutex> lock(mutex_);
        if (!mem_->empty()) {
            // 将当前 MemTable 标记为不可变，等待 Flush
            imm_ = mem_;
            has_imm_ = true;
            mem_ = std::make_shared<MemTable>();
        }
    }
    
    // Step 2: 等待所有 Flush/Compaction 完成
    {
        std::unique_lock<std::mutex> lock(mutex_);
        bg_cv_.wait(lock, [this] {
            return !has_imm_ && !bg_scheduled_;
        });
    }
    
    // Step 3: 通知后台线程退出
    shutting_down_ = true;
    bg_cv_.notify_all();
    if (bg_thread_.joinable()) bg_thread_.join();
    
    // Step 4: 关闭 WAL（截断到实际写入位置）
    if (wal_) {
        wal_->Sync();
        wal_->Close();
    }
    
    // Step 5: 可选 - 同步所有 SSTable 文件
    SyncAllSSTables();
}
```

### 3.2 WAL 截断与空间回收

#### 3.2.1 当前问题

当前 WAL 在 MemTable Flush 后**不会截断**：

```cpp
void DBImpl::FlushMemTable() {
    // ... 写入 L0 SSTable 完成 ...
    {
        // 重新创建 WAL（覆盖旧文件）
        wal_->Close();
        wal_ = std::make_unique<WALWriter>(wal_path);
        wal_->Open();
    }
}
```

这意味着：
- Flush 后旧 WAL 内容仍然存在，下次恢复时需**全量重放**
- 长时间运行后 WAL 恢复时间线性增长
- 尤其生产环境中数据库运行数天/周，恢复时间可能是**分钟级**

#### 3.2.2 设计方案

##### 按文件号管理 WAL

```cpp
class DBImpl : public DB {
    // 当前 WAL 编号（单调递增）
    uint64_t current_wal_number_ = 0;
    // 已 Flush 的最大序列号
    uint64_t flushed_seq_ = 0;
};
```

Flush 时：

```cpp
void DBImpl::FlushMemTable() {
    // Step 1: 在 Flush 前记录 imm_ 的最大序列号
    uint64_t imm_max_seq;
    {
        std::shared_lock lock(rw_mutex_);
        imm_max_seq = imm_->MaxSeq();  // 需要新增接口
    }
    
    // Step 2: Flush 到 SSTable
    WriteLevel0Table(mem_to_flush, &file_id);
    
    // Step 3: 更新已 Flush 的序列号
    {
        std::unique_lock lock(rw_mutex_);
        flushed_seq_ = std::max(flushed_seq_, imm_max_seq);
        
        // Step 4: 回收可以被安全删除的 WAL
        // 条件：WAL 中所有记录的 seq ≤ flushed_seq_
        TrimWAL();
    }
}
```

初始化时，只重放 `seq > flushed_seq_` 的记录：

```cpp
Status DBImpl::Initialize() {
```

## 4. P2 — 增强级优化

### 4.1 快照与事务支持

#### 4.1.1 设计方案

当前已有序列号机制和快照读（`snapshot_seq`），但缺少事务 API。

```cpp
class Transaction {
public:
    Transaction(DBImpl* db, uint64_t snapshot_seq);
    
    Status Put(const Slice& key, const Slice& value);
    Status Delete(const Slice& key);
    Status Get(const Slice& key, std::string* value);
    
    Status Commit();    // 原子提交所有变更
    void Rollback();    // 放弃所有变更
    
private:
    DBImpl* db_;
    uint64_t snapshot_seq_;
    bool committed_;
    
    // 写缓冲（写入在 Commit 时一次性提交）
    struct WriteOp {
        enum Type { kPut, kDelete };
        Type type;
        std::string key;
        std::string value;
    };
    std::vector<WriteOp> write_buf_;
};
```

#### 4.1.2 实现要点

- **原子提交**：`Commit()` 时在一个 WAL 记录中标记事务边界
- **快照隔离**：`Get()` 在 `snapshot_seq` 上读取，保证一致性
- **冲突检测**：在 `Commit()` 时检查 key 是否被其他事务修改

### 4.2 Range Delete（范围删除）

```cpp
// TableBuilder::FlushDataBlock
void TableBuilder::FlushDataBlock() {
    Slice raw_data = data_block_.Finish();
    
    std::string compressed;
    if (options_.compression != kNoCompression) {
        // 根据类型选择压缩算法
        // Snappy: 压缩率 ~2x, 速度 ~250MB/s
        // LZ4: 压缩率 ~1.8x, 速度 ~500MB/s
        Compress(raw_data, &compressed, options_.compression);

        if (compressed.size() < raw_data.size()) {
            // 在 BlockHandle 中标记已压缩
            info.handle.is_compressed = true;
            raw_data = Slice(compressed);
        }
    }
    writer_->Append(raw_data);
}
```

## 架构概览

LightKV v2.0-rc 支持两种运行模式：

### 单机模式（Standalone）

```
┌──────────────────────────────────────┐
│            lightkv_server            │
│  ┌─────────┐  ┌───────────────────┐  │
│  │ RESP    │  │ 存储引擎           │  │
│  │ 协议层  │◀─│ LSM-Tree + VLog   │  │
│  └─────────┘  │ Watch + ZSet 索引 │  │
│               └───────────────────┘  │
│  Port: 6379 (RESP)                   │
│  HTTP: 8080 (/metrics)               │
└──────────────────────────────────────┘
```

### 集群模式（Raft Cluster）

```
┌──────────┐   ┌──────────┐   ┌──────────┐
│ Node 1   │   │ Node 2   │   │ Node 3   │
│ RESP:6379│◀─▶│ RESP:6379│◀─▶│ RESP:6379│
│ Raft:16379  │ Raft:16379  │ Raft:16379  │
│ Raft Peer   │ Raft Peer   │ Raft Peer   │
└──────────┘   └──────────┘   └──────────┘
      │              │              │
      └──────────────┴──────────────┘
                Client
           (任意节点均可访问)
```

## 集群部署

### 系统要求

| 项目 | 最低要求 | 推荐配置 |
|------|---------|---------|
| CPU | 2 核 | 8 核+ |
| 内存 | 4GB | 16GB+ |
| 磁盘 | 20GB SSD | 100GB+ NVMe |
| 操作系统 | Linux 5.x+ | Ubuntu 22.04 / CentOS 9 |
| 网络 | 1Gbps | 10Gbps |

### 单机部署

```bash
# 编译
git clone https://github.com/your-org/lightkv.git
cd lightkv
mkdir build && cd build
cmake .. -DCMAKE_BUILD_TYPE=Release -DLIGHTKV_BUILD_TESTS=OFF
make -j$(nproc)

# 启动
./lightkv_server \
    --db-path /data/lightkv \
    --tcp-port 6379 \
    --http-port 8080 \
    --worker-threads 4

# 验证
redis-cli -p 6379 PING
# → +PONG
```

## 配置参考

### 存储引擎配置

| 参数 | 默认值 | 生产推荐 | 说明 |
|------|-------|---------|------|
| `memtable_size` | 64MB | 256MB | 提高减少 flush 频率，增大内存占用 |
| `block_cache_size` | 512MB | 2GB+ | 根据可用内存的 20% 设置 |
| `block_size` | 4KB | 16KB | 增大减少索引开销，增加读放大 |
| `value_threshold` | 64KB | 64KB | 超过此值自动分离到 vlog |
| `vlog_file_size_limit` | 512MB | 1GB | vlog 文件滚动大小 |
| `compaction_rate_limit` | 0（不限速） | 500MB/s | 限制 compaction 带宽，避免影响前台 |

### Raft 配置

| 参数 | 默认值 | 说明 |
|------|-------|------|
| `--cluster` | false | 启用 Raft 集群模式 |
| `--node-id` | 0 | 本节点 ID（1-255，集群内唯一） |
| `--raft-host` | 0.0.0.0 | Raft 内部通信监听地址 |
| `--raft-port` | 16379 | Raft 内部通信端口 |
| `--raft-peers` | — | 集群节点列表 |
| `--election-timeout` | 150ms | 选举超时最小值（建议 150-300ms） |
| `--heartbeat-interval` | 50ms | Leader 心跳间隔 |

### Server 配置

| 参数 | 默认值 | 生产推荐 | 说明 |
|------|-------|---------|------|
| `--worker-threads` | 0（单线程） | CPU 核数 | 多线程处理命令 |
| `--max-conns` | 1024 | 4096 | 最大连接数 |
| `--slowlog-threshold-ms` | 100ms | 50ms | 慢查询日志阈值 |
| `--requirepass` | 空 | 必填 | 认证密码 |

## 监控与告警

### Prometheus 指标

通过 HTTP `/metrics` 端点获取（需编译时启用 `LIGHTKV_METRICS`）：

```prometheus
# HELP lightkv_put_total Total number of Put calls
# TYPE lightkv_put_total counter
lightkv_put_total 123456
# HELP lightkv_get_latency_ms Get latency in milliseconds
# TYPE lightkv_get_latency_ms histogram
lightkv_get_latency_ms{quantile="0.5"} 0.12
lightkv_get_latency_ms{quantile="0.99"} 1.85
# HELP lightkv_memtable_bytes MemTable memory usage
# TYPE lightkv_memtable_bytes gauge
lightkv_memtable_bytes 67108864
# HELP lightkv_disk_usage_bytes Disk usage
# TYPE lightkv_disk_usage_bytes gauge
lightkv_disk_usage_bytes 2147483648
```

### Prometheus 告警规则

```yaml
groups:
  - name: lightkv
    rules:
      - alert: LightKVHighLatency
        expr: lightkv_get_latency_ms{quantile="0.99"} > 10
        for: 5m
        annotations:
          summary: "LightKV 99th percentile latency > 10ms"

      - alert: LightKVHighDiskUsage
        expr: lightkv_disk_usage_bytes > 85
        for: 10m
        annotations:
          summary: "LightKV disk usage > 85%"

      - alert: LightKVNoLeader
        expr: lightkv_raft_leader_id == 0
        for: 10s
        annotations:
          summary: "Raft cluster has no leader"

      - alert: LightKVNodeDown
        expr: up{job="lightkv"} == 0
        for: 30s
        annotations:
          summary: "LightKV node is down"
```

## 容量规划

### 内存估算

```
总内存 ≈ block_cache_size + memtable_size + 系统开销(约 10%)

示例（16GB 内存服务器）：
- block_cache_size: 4GB  （25%）
- memtable_size: 512MB   （3%）
- 系统开销/其他: 1.5GB   （9%）
- 剩余: 约 10GB 用于 OS Page Cache
```

### 磁盘估算

```
磁盘占用 = 数据总量 × 写放大因子

LSM-Tree 写放大：
- Level 0: ~4x
- Level 1-6: ~10x each
- 平均写放大: ~3-5x（取决于 compaction 策略）

大 Value 场景：
- Value ≤ 64KB: 走 LSM-Tree，写放大 ~3-5x
- Value > 64KB: 走 VLog，写放大 ~1.1x（仅 GC）

示例（100GB 数据）：
- 小 Value（<64KB）: ~300-500GB 磁盘
- 大 Value（>64KB）: ~110-150GB 磁盘
```

### Raft 日志清理

Raft 日志默认存储在 `{db_path}/raft_state.dat`，单节点日志大小限制为 64MB。建议将数据目录放在单独的 SSD 分区上。

## 故障恢复

### Leader 故障

当 Leader 节点宕机时：
1. 其他 Follower 检测到选举超时（默认 150-300ms）
2. 发起选举，拥有最新的日志的节点当选
3. 新的 Leader 提交一条 NoOp 日志宣告自己就绪
4. 恢复时间：通常 300-500ms

**客户端行为**：客户端应配置连接到所有节点，在 Leader 故障时切换。建议 SDK 使用连接池 + 自动重连（Phase 2 已实现）。

### Follower 故障

1. Follower 恢复后重新加入集群
2. Leader 检测到并补发缺失的日志条目
3. 如果落后太多，通过安装快照（InstallSnapshot）同步
4. 恢复后正常提供服务（只读查询）

### 节点永久故障

如果节点不可恢复（磁盘损坏等）：

1. 使用 `CLUSTER FORGET` 从集群中移除故障节点
2. 添加新节点替换
3. 新节点从 Leader 同步完整数据

### 数据文件损坏

```bash
# 检查数据目录
ls -la /data/lightkv/

# 尝试重建
# 如果 WAL 损坏，可尝试删除 WAL 仅用 MANIFEST 恢复
# （注意：会丢失未 flush 的数据）
mv /data/lightkv/wal.log /data/lightkv/wal.log.bak
./lightkv_server --db-path /data/lightkv
```

## 性能调优

### 编译优化

```bash
# Release 模式（默认 -O3 -march=native）
cmake .. -DCMAKE_BUILD_TYPE=Release

# 禁用 LZ4（减少依赖，轻微影响压缩）
cmake .. -DLZ4_INCLUDE_DIR= -DLZ4_LIBRARY=
```

### 操作系统调优

```bash
# 增大 net.core.somaxconn
echo 4096 > /proc/sys/net/core/somaxconn

# 启用 TCP 快速回收（Linux 4.12+ 推荐 tcp_fastopen）
echo 3 > /proc/sys/net/ipv4/tcp_fastopen

# 增大文件描述符限制
ulimit -n 65536

# 使用 deadline/noop I/O 调度器（SSD）
echo deadline > /sys/block/nvme0n1/queue/scheduler
```

### 关键性能参数

| 场景 | 调整方案 | 预期收益 |
|------|---------|---------|
| 大量小 Value（<1KB） | 增大 block_size 到 16KB | 减少索引开销 30% |
| 大量大 Value（>64KB） | 降低 value_threshold 到 32KB | 减少 compaction 写放大 |
| 写入密集型 | 增大 memtable_size 到 512MB | 减少 flush 频率 |
| 读取密集型 | 增大 block_cache_size | 提高缓存命中率 |
| 混合负载 | 设置 compaction_rate_limit | 减少写入延迟抖动 |

### 基准测试

```bash
# 使用自带 bench 工具
./bench_cpp

# 使用 redis-benchmark（标准 Redis 基准测试工具）
redis-benchmark -p 6379 -n 100000 -c 50 -t SET,GET

# 使用 Python SDK 基准测试
cd clients/python
python tests/bench.py
```

## 升级指南

### 滚动升级（集群模式）

1. 停止节点 3 的 lightkv_server
2. 替换二进制文件
3. 启动节点 3，等待恢复
4. 依次升级节点 2、节点 1

### 数据迁移

使用 `lightkv_migrate` 工具从现有 Redis 迁移：

```bash
# 全量迁移
./lightkv_migrate \
    --redis-host 192.168.1.100 --redis-port 6379 \
    --target-host 192.168.1.10 --target-port 6379

# 指定前缀迁移
./lightkv_migrate \
    --redis-host 192.168.1.100 --redis-port 6379 \
    --target-host 192.168.1.10 --target-port 6379 \
    --prefix "session:"

# 带密码认证
./lightkv_migrate \
    --redis-host 192.168.1.100 --redis-port 6379 \
    --redis-auth "your-password" \
    --target-host 192.168.1.10 --target-port 6379
```

## 附录：Docker 部署

### Dockerfile

```dockerfile
FROM ubuntu:22.04 AS build

RUN apt-get update && apt-get install -y \
    build-essential cmake git liblz4-dev

WORKDIR /build
COPY . .
RUN mkdir build && cd build && \
    cmake .. -DCMAKE_BUILD_TYPE=Release -DLIGHTKV_BUILD_TESTS=OFF && \
    make -j$(nproc) lightkv_server lightkv_migrate

FROM ubuntu:22.04
RUN apt-get update && apt-get install -y liblz4-1 && rm -rf /var/lib/apt/lists/*
COPY --from=build /build/build/lightkv_server /usr/local/bin/
COPY --from=build /build/build/lightkv_migrate /usr/local/bin/

EXPOSE 6379 8080 16379

ENTRYPOINT ["lightkv_server"]
```

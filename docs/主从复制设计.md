# LightKV 主从复制设计文档

## 1. 概述

### 1.1 目标

实现基于命令流的主从复制（Master-Slave Replication），使 LightKV 具备以下能力：
- **数据冗余**：Master 的写操作自动同步到 Slave，防止单机故障导致数据丢失
- **读写分离**：Slave 可处理读请求，分担 Master 压力
- **故障恢复基础**：为后续哨兵自动故障转移提供数据基础

### 1.2 设计原则

- **命令流复制**：Master 将写命令序列化为 RESP 格式，Slave 接收并回放
- **全量 + 增量**：Slave 首次连接时全量同步（RDB 快照），之后增量同步（命令流）
- **异步复制**：Master 不等待 Slave 确认，保证写入性能
- **断线重连**：Slave 断线后自动重连，支持部分重同步（PSYNC）
- **最小侵入**：复用现有 RESP 协议和命令处理框架

## 2. 架构设计

### 2.1 角色定义

```
┌─────────────────┐         命令流 (RESP)        ┌─────────────────┐
│    Master       │ ──────────────────────────►   │    Slave        │
│                 │                              │                 │
│ - 接受客户端写  │                              │ - 只读模式       │
│ - 接受客户端读  │                              │ - 回放命令流     │
│ - 广播写命令    │                              │ - 可接受读请求   │
│ - 维护 repl_id  │                              │ - 维护 offset    │
│ - 维护 backlog  │                              │                 │
└─────────────────┘                              └─────────────────┘
```

### 2.2 核心概念

| 概念 | 说明 |
|------|------|
| `repl_id` | 复制 ID，Master 启动时生成的 UUID，标识当前复制流 |
| `repl_offset` | 复制偏移量，Master 和 Slave 各自维护的命令流字节偏移 |
| `backlog` | Master 维护的环形缓冲区，存储最近发送的命令流，用于部分重同步 |
| `RDB 快照` | Master 当前数据的全量快照，用于 Slave 首次全量同步 |

### 2.3 复制状态机

```
Slave 状态:
  HANDSHAKE → 发送 PING 确认 Master 可达
      │
      ▼
  WAIT_BGSAVE → 等待 Master 生成 RDB 快照
      │
      ▼
  FULL_SYNC → 接收 RDB 快照并加载
      │
      ▼
  WAIT_REPL → 等待增量命令流
      │
      ▼
  ONLINE → 正常接收增量命令流
      │
      ▼
  (断线) → DISCONNECTED → 尝试 PSYNC → 成功 → ONLINE
                              │
                              ▼ (PSYNC 失败)
                              FULL_SYNC (重新全量同步)
```

## 3. 协议设计

### 3.1 复制专用命令

复用 RESP 协议，在 Master 和 Slave 之间使用内部命令：

| 命令 | 方向 | 说明 |
|------|------|------|
| `REPLCONF listening-port <port>` | Slave→Master | Slave 告知监听端口 |
| `REPLCONF capa <psync2>` | Slave→Master | 声明支持的能力 |
| `PSYNC <repl_id> <offset>` | Slave→Master | 请求部分/全量同步 |
| `FULLRESYNC <repl_id> <offset>` | Master→Slave | 全量同步响应 |
| `CONTINUE <repl_id>` | Master→Slave | 部分重同步成功 |
| `PING` | Master→Slave | 心跳保活 |
| `SELECT <db>` | Master→Slave | 切换数据库（预留） |
| 写命令 (SET/DEL/SETEX...) | Master→Slave | 实际数据同步 |

### 3.2 PSYNC 流程

```
Slave: PSYNC ? -1          (首次连接，repl_id=? offset=-1)
Master: FULLRESYNC <uuid> 0  (全量同步，返回新 repl_id 和起始 offset)
       [RDB 快照数据流]
       [增量命令流...]

Slave: PSYNC <uuid> 12345  (断线重连，携带上次 offset)
Master: CONTINUE <uuid>     (部分重同步，从 offset 12345 继续)
       [增量命令流...]

Master: CONTINUE <uuid>     (backlog 中无 offset 12345，要求全量)
       → 实际返回 FULLRESYNC 重新全量同步
```

### 3.3 命令流格式

Master 将每个写命令序列化为 RESP 格式，追加到复制流：

```
*3\r\n$3\r\nSET\r\n$3\r\nfoo\r\n$3\r\nbar\r\n   ← SET foo bar
*2\r\n$3\r\nDEL\r\n$3\r\nfoo\r\n                 ← DEL foo
*4\r\n$5\r\nSETEX\r\n$3\r\nkey\r\n$1\r\n5\r\n$5\r\nhello\r\n  ← SETEX key 5 hello
```

每个命令前附加一个偏移量标记（内部使用，不发送到 Slave）：

```
[repl_offset: 1024] *3\r\n$3\r\nSET\r\n$3\r\nfoo\r\n$3\r\nbar\r\n
```

### 4.2 Master 端组件

```cpp
class ReplicationMaster {
public:
    ReplicationMaster(const ServerOptions& opts);
```

### 4.4 复制流注入点

在 Server 的命令处理层注入复制逻辑：

```cpp
// server.cpp - Impl::process_command()
void process_command(Connection& conn, const std::vector<std::string>& args) {
    std::string cmd = to_upper(args[0]);

    // 写命令需要复制到 Slave
    if (is_write_command(cmd)) {
        std::string resp_cmd = serialize_resp(args);
        repl_master_->FeedReplicationStream(resp_cmd);
    }

    // 执行命令
    std::string reply = dispatch_command(args);
    send_reply(conn, reply);
}
```

### 4.5 全量同步（RDB 快照）

```
Master 端:
  1. 收到 PSYNC ? -1 → 触发 BGSAVE
  2. BGSAVE 完成后，读取 RDB 文件
  3. 发送 "$<size>\r\n" + RDB 数据
  4. 发送 backlog 中积累的增量命令

Slave 端:
  1. 接收 RDB 数据，写入临时文件
  2. 接收完毕 → FLUSHDB 清空本地数据
  3. 加载 RDB 快照到本地 DB
  4. 开始接收并回放增量命令
```

RDB 格式复用现有 DB 导出能力（通过 Scan 全量读取 + 序列化）：

```
RDB 文件格式:
┌──────────┬──────────────────────┬──────────┐
│ Header   │ KV Entries (批量)     │ Footer   │
│ "LKVDB01"│ [key_len|key|val_len|val]... │ CRC32 │
└──────────┴──────────────────────┴──────────┘
```

### 4.6 部分重同步（PSYNC）

```cpp
// Master 处理 PSYNC
std::string ReplicationMaster::HandlePSYNC(
    const std::string& slave_repl_id,
    uint64_t slave_offset,
    int slave_fd) {

    if (slave_repl_id == "?" || slave_repl_id != repl_id_) {
        // 首次连接或 repl_id 不匹配 → 全量同步
        uint64_t start_offset = repl_offset_;
        return resp_bulk_string(
            "FULLRESYNC " + repl_id_ + " " + std::to_string(start_offset));
    }

    // 检查 backlog 中是否有 slave_offset
    if (backlog_.Contains(slave_offset)) {
        return resp_bulk_string("CONTINUE " + repl_id_);
    }

    // backlog 已覆盖不到 → 全量同步
    return resp_bulk_string(
        "FULLRESYNC " + repl_id_ + " " + std::to_string(repl_offset_));
}
```

### 4.7 环形缓冲区（Backlog）

```cpp
class ReplBacklog {
public:
    ReplBacklog(size_t size);

    // 追加数据到 backlog
    void Append(const std::string& data, uint64_t offset);

    // 获取从指定 offset 开始的数据
    std::string GetFrom(uint64_t offset) const;

    // 检查 offset 是否在 backlog 范围内
    bool Contains(uint64_t offset) const;

    uint64_t FirstOffset() const { return first_offset_; }
    uint64_t LastOffset() const { return last_offset_; }

private:
    std::vector<char> buffer_;    // 环形缓冲区
    uint64_t first_offset_;       // 缓冲区中最早的偏移量
    uint64_t last_offset_;        // 缓冲区中最新的偏移量
    size_t size_;                 // 缓冲区大小
};
```

## 6. 线程模型

```
Master:
  ┌─────────────────────────────────────────┐
  │ Event Loop (kqueue/epoll)               │
  │  ├── Client connections (读/写)          │
  │  ├── Slave connections (发送复制流)      │
  │  └── HTTP monitor                       │
  │                                         │
  │ 写命令处理:                              │
  │  1. 执行 db_->Put()                     │
  │  2. FeedReplicationStream()             │
  │  3. 标记所有 Slave conn 为 writable      │
  │  4. Event Loop 发送 backlog 数据         │
  └─────────────────────────────────────────┘

Slave:
  ┌─────────────────────────────────────────┐
  │ Event Loop (kqueue/epoll)               │
  │  ├── Client connections (只读)           │
  │  ├── Master connection (接收复制流)      │
  │  └── HTTP monitor                       │
  │                                         │
  │ 复制流处理:                              │
  │  1. 从 Master conn 读取 RESP 命令        │
  │  2. 解析命令                             │
  │  3. 直接调用 db_->Put()/Delete()         │
  │     (不走认证/权限检查)                   │
  │  4. 更新 repl_offset                     │
  └─────────────────────────────────────────┘
```

## 7. 故障场景处理

### 7.1 Slave 断线重连

```
1. Slave 检测到 Master 连接断开
2. 进入 DISCONNECTED 状态
3. 定时重试连接（指数退避：1s, 2s, 4s, ... 最大 60s）
4. 连接成功后发送 PSYNC <repl_id> <offset>
5. Master 检查 backlog:
   - 包含 offset → CONTINUE，发送增量数据
   - 不包含 offset → FULLRESYNC，重新全量同步
```

### 7.2 Master 重启

```
1. Master 重启后生成新的 repl_id
2. 所有 Slave 的 PSYNC 会收到 FULLRESYNC
3. Slave 重新执行全量同步流程
4. 旧数据被 FLUSHDB 清空，加载新快照
```

### 7.3 Slave 重启

```
1. Slave 重启后保留本地 repl_id 和 offset（持久化到本地文件）
2. 自动连接 Master 并发送 PSYNC
3. 如果 offset 仍在 backlog 中 → 增量同步
4. 否则 → 全量同步
```

### 7.4 网络分区

```
1. 网络恢复后 Slave 自动重连
2. PSYNC 判断是否需要全量同步
3. 如果 Master 已重启 → 全量同步
4. 如果 Master 未重启且 offset 有效 → 增量同步
```

## 9. 性能影响评估

| 场景 | 影响 | 说明 |
|------|------|------|
| 无 Slave 连接 | ~0% | 复制流不激活，无额外开销 |
| 1 个 Slave | < 2% | 命令序列化 + 发送，异步不阻塞 |
| N 个 Slave | < 2% * N | 每个 Slave 独立发送，共享 backlog |
| Slave 断线重连 | 取决于 backlog | 命中 backlog 则毫秒级恢复 |
| 全量同步 | 取决于数据量 | RDB 快照传输期间 Master 不受影响 |

## 未修复（性能优化，非阻塞）

### 1. ZSet 所有操作全量扫描排序 O(N log N)

- **文件**: `src/server.cpp:2601-2631` (`zscan_all`)
- **现象**: `ZRANGE`、`ZREVRANGE`、`ZCOUNT`、`ZRANK` 等所有 ZSet 命令都调用 `zscan_all` 进行全量 Scan + 内存排序。时间复杂度 O(N log N)，对大 ZSet（如排行榜百万级成员）极其低效。
- **影响**: 性能瓶颈，ZSet 操作延迟随数据量线性增长。
- **修复方向**: 引入跳表或排序数组作为 ZSet 索引，支持 O(log N) 的 rank/range 操作。
- **复杂度**: 高（需要重新设计 ZSet 存储结构）

### 2. SkipList 每次 Insert/Find 将 Slice 转 std::string

- **文件**: `include/lightkv/skiplist.h:341-361`
- **现象**: 公共包装类 `SkipList` 的 `Insert`、`Find`、`Contains`、`SeekGE` 每次调用都将 `Slice` 转换为 `std::string`（堆分配），在热路径上引入大量不必要的内存分配。
- **影响**: 性能开销，MemTable 操作每次额外分配 2 次堆内存（key + value）。
- **修复方向**: 使用 `std::string_view` + 自定义比较器，或让内部 SkipList 直接支持 `Slice` 类型。
- **复杂度**: 中（需要修改 SkipList 模板参数和比较器）

## 备注

- 以上 2 个问题均为**性能优化**，不影响功能正确性。
- 当前实现在中小数据量下表现良好，仅在大数据量场景下需要优化。
- 可根据实际业务需求决定是否优化。

## 未修复（需架构级重构）

### 1. Scan Iterator 锁释放后访问可能已删除的 SSTable

- **文件**: `src/iterator.cpp:253-283`, `src/db_impl.cpp:249-272`
- **现象**: `DBImpl::Scan` 创建 `Iterator` 后释放 `rw_mutex_`。Iterator 内部持有 `SSTable*` 裸指针，若 compaction 在锁释放后删除了该 SSTable，后续 `Next()` 调用会访问已释放内存。
- **影响**: use-after-free，可能崩溃或数据损坏。
- **修复方向**: 为 SSTable 引入引用计数（shared_ptr），Iterator 持有 shared_ptr 保证生命周期。或在 Iterator 存活期间禁止删除其引用的 SSTable 文件。
- **复杂度**: 高（涉及 DB/Iterator/SSTable/Compaction 四个模块交互）

### 2. Compaction 丢弃快照需要的旧版本

- **文件**: `src/compaction.cpp:120-141`
- **现象**: `CompactionWorker` 合并时只保留每个 key 的最新版本，丢弃所有旧版本。但如果某个旧版本对活跃 snapshot 仍然可见（`old_seq <= snapshot_seq`），删除它会导致快照读取失败。
- **影响**: 违反 MVCC 语义，快照读可能返回 NotFound 或错误值。
- **修复方向**: 维护活跃快照列表（`DBImpl::snapshots_`），Compaction 时为每个 key 保留至少一个对最老活跃 snapshot 可见的版本。

### 3. ZSet 所有操作全量扫描排序 O(N log N)

- **文件**: `src/server.cpp:2601-2631` (`zscan_all`)
- **现象**: `ZRANGE`、`ZREVRANGE`、`ZCOUNT`、`ZRANK` 等所有 ZSet 命令都调用 `zscan_all` 进行全量 Scan + 内存排序。时间复杂度 O(N log N)，对大 ZSet（如排行榜百万级成员）极其低效。
- **影响**: 性能瓶颈，ZSet 操作延迟随数据量线性增长。
- **修复方向**: 引入跳表或排序数组作为 ZSet 索引，支持 O(log N) 的 rank/range 操作。

### 4. SkipList 每次 Insert/Find 将 Slice 转 std::string

- **文件**: `include/lightkv/skiplist.h:341-361`
- **现象**: 公共包装类 `SkipList` 的 `Insert`、`Find`、`Contains`、`SeekGE` 每次调用都将 `Slice` 转换为 `std::string`（堆分配），在热路径上引入大量不必要的内存分配。
- **影响**: 性能开销，MemTable 操作每次额外分配 2 次堆内存（key + value）。
- **修复方向**: 使用 `std::string_view` + 自定义比较器，或让内部 SkipList 直接支持 `Slice` 类型。

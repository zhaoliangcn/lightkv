## 1. 定位与愿景 {#positioning}

### 1.1 核心定位

> **Redis 协议兼容，持久化存储，可嵌入可分布式。**
> 用户不需要改一行代码，把 Redis 换成 LightKV，内存省一半。

### 1.2 设计原则

1. **协议优先** — 所有新增功能不影响 Redis 协议兼容性，现有客户端零修改
2. **分层可裁剪** — 核心引擎层、网络层、分布式层、SDK 层独立，嵌入式场景可裁剪掉网络层
3. **生产级第一** — 不如 Redis 快可以接受，但不能不如 Redis 可靠
4. **大 Value 友好** — 突破 Redis 的"全内存"天花板，数据量超内存时自动降冷

### 1.3 非目标（v2.0 不做的事）

为避免范围蔓延，v2.0 显式排除以下内容：

- **向量索引** — 不做 HNSW/IVF，AI 推理场景的向量搜索推迟到 v3.0
- **图数据库查询能力** — 不做图遍历、最短路径等图查询原语
- **Redis Stream 消费者组的完整语义** — 仅做 RESP 协议兼容响应，不实现消费者组持久化与重平衡
- **Redis Module API** — 不支持加载第三方 Redis Module，工程量过大且偏离核心定位
- **多语言 UDF / 服务端脚本** — 不集成 Lua/JavaScript 服务端脚本引擎（Redis 的 `EVAL` 仅做协议响应，不实际执行）
- **跨数据中心多活** — Raft 共识仅覆盖单集群内的一致性，不做跨地域多活仲裁

## 4. 存储引擎生产级加固 {#engine-hardening}

### 4.1 当前问题

| 问题 | 风险 | 影响范围 |
|------|------|---------|
| WAL 用 mmap，crash 时最后写入可能丢失 | 数据丢失 | 所有场景 |
| Compaction 无限速，可能阻塞写入 | 写入延迟抖动 | 高吞吐场景 |
| Bloom Filter 每 SSTable 独立加载 | 读放大 | 大数据量场景 |
| 没有 MemTable 内存上限保护 | OOM | 大 Value 写入 |

### 4.2 改造方案

#### 4.2.1 WAL Crash Consistency

```cpp
// 当前：
// mmap + msync，crash 时 msync 不保证数据落盘

// v2.0：
// 每次 Append 写入后，在 write_pos_ 位置写入一个 4 字节的 checksum
// 恢复时从末尾向前扫描，找到最后一个完整的记录
// 支持 WAL 记录级别的原子性：要么全部写入，要么全部丢弃
```

**具体变更**：

1. 每条 WAL 记录尾部追加 4 字节 CRC32C（已有前置校验，改为尾部校验）
2. 恢复时从文件末尾向前扫描，跳过不完整的最后一条记录
3. 每次 Sync 在文件末尾写入一个 8 字节的 `sync marker`（单调递增的 log sequence number）
4. 恢复时如果 sync marker 缺失，则丢弃其后所有记录

**seq 分配时机与空洞语义**（关键澄清）：

- `last_seq_` 在 `Put`/`Delete` 入口即 `fetch_add` 分配，**早于** WAL 写入
- sync marker 存储的是"已 sync 到的 last_seq"，恢复时从 marker 恢复
- marker 之后已分配但未 sync 的 seq 会**永久成为空洞**——这些 seq 对应的 WAL 记录被丢弃，但 seq 数已前进
- 这是安全语义：seq 单调递增，空洞不影响正确性（只是浪费 seq 数）
- MemTable 不重放丢弃的记录，因此不会出现"重复 seq"

**崩溃场景示例**：

```
seq=100  Put("a", "1")   ← sync marker 在此（last_sync_seq=100）
seq=101  Put("b", "2")   ← 已写入 MemTable，未 sync
seq=102  Put("c", "3")   ← 已写入 MemTable，未 sync
[CRASH]

恢复行为：
1. 读取 sync marker → last_sync_seq=100
2. 丢弃 seq=101、102 的 WAL 记录
3. last_seq_ 恢复为 100
4. 下次分配从 seq=103 开始（101、102 永久空洞）
5. MemTable 中无 "b"、"c"（因 WAL 丢弃，不重放）
```

#### 4.2.2 Compaction 限速

```cpp
// 新增 RateLimiter 类
class RateLimiter {
    // 基于 token bucket 算法
    // 限制 compaction 的写入带宽，留出资源给前台读写
    void Request(size_t bytes);  // 阻塞直到 token 可用
};

// CompactionWorker 中调用
void DoCompaction() {
    // 每写入一个 data block 前调用 rate_limiter_.Request()
    rate_limiter_->Request(block_size);
}
```

**配置项**：

```cpp
struct Options {
    // 新增
    uint64_t compaction_rate_limit = 0;  // 0 = 不限速，单位 bytes/sec
    bool sync_wal_every_write = true;    // 每次写入是否 sync WAL
    size_t max_memtable_size = 128 * 1024 * 1024;  // MemTable 硬上限
};
```

#### 4.2.3 Bloom Filter 缓存

当前：每个 SSTable 打开时加载 Bloom Filter 数据。

v2.0：将 Bloom Filter 数据放入 Block Cache 统一管理，按 LRU 淘汰。

```cpp
// 在 SSTable::Open() 中
// 当前：bloom_filter_ = BloomFilter(data_ptr, size);
// v2.0：bloom_data_ = block_cache_->LookupOrInsert(file_id, kBloomBlockIndex, ...);
```

## 5. 原子批量操作（事务基础） {#batch-atomic}

### 5.1 设计目标

- 支持 **单次 WAL 写入包含多个 key 变更**
- 要么全部写入，要么全部丢弃（原子性）
- 不阻塞其他写入（非锁式）

**隔离性声明**（重要边界）：

BatchWrite 提供**原子性但不提供串行化隔离**：

- WAL 写入是原子的（单次 mmap 写，crash 后要么全部可见要么全部不可见）
- MemTable 写入是**逐步**的——每个 key 单独 `Insert`，期间其他线程的 `Get` 可能看到部分变更
- 不持有任何全局锁，吞吐与单次 `Put` 一致

如需串行化隔离（读写都看不到中间态），调用者应使用外层 `mutex_`：

```cpp
{
    std::lock_guard<std::mutex> lock(db_mutex);
    db->BatchWrite(ops);  // 持锁期间其他 Get 看不到部分变更
}
```

**为什么不内置串行化**：内置会强制所有 BatchWrite 持有全局锁，退化到串行吞吐，违背"非锁式"目标。把隔离决策权交给调用者，普通场景拿吞吐，强一致场景拿锁。

### 5.2 WAL 格式扩展

```
当前 WAL 记录格式：
[ CRC(4) | Length(4) | Type(1) | Key... | Value... | Seq(8) ]

v2.0 批量格式：
[ CRC(4) | Length(4) | kTypeBatch(1) | Count(4) |
  [ Type(1) | Key... | Value... ] × Count |
  Seq(8) ]
```

### 5.3 API 扩展

```cpp
// 存储引擎层
class DBImpl {
    // 原子批量写入
    Status BatchWrite(const std::vector<WriteOp>& ops);
    
    // 带 CAS 条件的写入
    Status CompareAndSwap(const Slice& key, const Slice& expected, const Slice& value);
};
```

### 5.4 上层使用

```cpp
// 分布式文件系统重命名目录：
// 需要原子更新：删除 old_dir_key，创建 new_dir_key
db->BatchWrite({
    {kDelete, "dir:/old/name", ""},
    {kPut,    "dir:/new/name", metadata}
});

// 分布式锁释放：
// 只有持有锁的客户端才能删除
db->CompareAndSwap("lock:mykey", "my-session-id", "");
```

## 6. Watch 机制 {#watch}

### 6.1 设计目标

- 客户端可以 Watch 一个 key 或 key 前缀
- 当 key 被 Put/Delete 时，Watcher 收到通知
- 类似 etcd 的 `Watch`，但接口兼容 Redis 的 Pub/Sub
- 嵌入式场景可裁剪掉

### 6.2 架构

```
┌──────────┐     ┌──────────────┐     ┌──────────┐
│ Watcher  │────▶│  WatchHub    │◀────│  DBImpl  │
│ (Client) │     │  (单例)      │     │ (写入时  │
│          │     │  key → []Watcher │     │  通知)   │
└──────────┘     └──────────────┘     └──────────┘
```

### 6.3 实现要点

```cpp
// WatchHub 核心逻辑
class WatchHub {
    // 订阅
    // WATCH key          → 精确匹配
    // WATCH key:*        → 前缀匹配
    void Subscribe(const std::string& pattern, Watcher* watcher);
    
    // 通知（由 DBImpl::Put/Delete 触发）
    void Notify(const std::string& key, ChangeType type);
    
    // 通知队列（避免阻塞写入路径）
    std::queue<Notification> pending_;
    std::thread notification_thread_;
};
```

### 6.4 命令设计

两条路径**并存但定位不同**，客户端库应按场景选择：

```
# 路径 A：Redis 兼容方式（Pub/Sub 扩展）
SUBSCRIBE __keyspace@0__:mykey     → 收到 key 变更通知
SUBSCRIBE __keyevent@0__:set       → 收到 SET 事件

  语义：单向广播，无保活，断线期间事件丢失
  定位：让现有 Redis 客户端的 keyspace notification 直接可用
  实现：WatchHub 在 Notify 时同时 PUBLISH 到 keyspace 频道

# 路径 B：增强命令（类似 etcd 风格）
WATCH mykey                        → 监听单个 key
WATCH mykey FROM <revision>        → 断线重连后补送漏掉的事件
WATCHPREFIX /dir/                  → 监听前缀
UNWATCH mykey                      → 取消监听

  语义：双向 RPC + revision 序号，支持断线重连补数据
  定位：分布式系统 / 文件系统元数据监听，不能容忍事件丢失
  实现：WatchHub 为每个 watcher 维持 last_pushed_revision，
        重连时从 revision log 补发
```

**两路径关系**：

- 路径 A 是路径 B 的**子集**——路径 B 的 watcher 也可以选择广播到 keyspace 频道
- 客户端库不应两条都调：普通 Redis 用户调 A，分布式场景调 B
- 路径 B 的 revision log 复用 WAL 的 seq（无需独立序号空间）

## 7. 大 Value 分离存储 {#large-value}

### 7.1 问题分析

当前 SSTable 的 data block 默认 4KB。如果 value 为 1MB：

- 写入时被分到 256 个 block 中（SSTable 的 data block 连续存储，Block Cache 命中时读放大为 0）
- **真正的问题在 compaction 阶段**：compaction 要把整个 1MB value 读出再写入新 SSTable，1MB value 在每次 compaction 中都被搬运一次
- **写放大**：1MB 的 value 在 LSM-Tree 的每一层都被重写一次，到 L6 共写 6 次 = 6MB 写磁盘
- 大 value 还会撑爆 Block Cache（一个 value 占 256 个 cache slot），挤占其他热数据

**关键澄清**：当前实现读放大并不严重（mmap + Block Cache 命中时读放大为 0，未命中时也只读一次连续 block）。真正的痛点是 **compaction 阶段的写放大**与 **Block Cache 污染**。

### 7.2 方案：Value 分离（WiscKey 风格）

```
写入路径：
1. Value > 64KB → 写入独立的 value log（vlog）
2. SSTable 中只存 value 的指针 (vlog_offset, vlog_length)
3. 读取时先查 SSTable 拿到指针，再读 vlog

读取路径：
1. Get(key) → SSTable 查到 value 指针（vlog_offset, vlog_length）
2. 从 vlog 文件中读取 value 数据
3. 返回给用户

Compaction 时：
- SSTable 的 compaction 不再搬运大 value（只搬运指针）
- 大 value 的垃圾回收由 vlog 独立处理（详见 7.5）
```

### 7.3 配置

```cpp
struct Options {
    // 新增
    size_t value_threshold = 64 * 1024;  // 超过此大小的 value 分离存储
    size_t vlog_file_size = 512 * 1024 * 1024;  // vlog 文件大小上限
    bool vlog_gc_enabled = true;  // 是否启用 vlog 垃圾回收
};
```

### 7.4 收益

下表针对 **compaction 写放大**（即同一 value 在 LSM-Tree 各层被重写的次数）与 **Block Cache 污染**：

| Value 大小 | 当前（compaction 写放大） | v2.0 大 Value 分离 | Block Cache 占用 |
|-----------|----------------------|-------------------|-----------------|
| 1KB       | 1x（不变）            | 1x（不变）         | 1 slot（不变）   |
| 1MB       | 每次 compaction 搬运 1MB | 只搬指针 16B       | 从 256 slot 降到 1 slot |
| 64MB      | 每次 compaction 搬运 64MB | 只搬指针 16B       | 从 16,384 slot 降到 1 slot |

**澄清**：当前实现的**读放大**因 mmap + Block Cache 命中并不显著；上表数值反映的是 **compaction 阶段的写放大**与 **Block Cache 污染**。v2.0 的分离存储主要解决这两类问题，而非读放大。

### 7.5 vlog 垃圾回收

分离存储的代价是 vlog 中会积累**已被覆盖或删除的旧 value**，需要独立 GC 回收磁盘空间。这是 WiscKey 论文中最棘手的部分，必须明确触发条件、回收过程与并发控制。

**触发条件**：

```
vlog 文件 usage < 50% 时启动 GC
（usage = 存活 value 字节数 / vlog 文件总大小）
触发频率：每完成一次 compaction 后检查一次
可选：手动 VLOG GC 命令强制触发
```

**回收过程**：

```
1. 选定一个 vlog 文件作为 GC 对象（最旧的一个）
2. 从头扫描 vlog，对每个 value：
   a. 查 SSTable 拿到该 key 当前 value 指针
   b. 比对指针：若 (vlog_offset, vlog_length) 与当前扫描位置一致 → 存活
   c. 不一致 → 已被覆盖/删除，跳过
3. 存活的 value 追加写入新 vlog（活动 vlog）末尾
4. 更新 SSTable 中对应 key 的指针（指向新 vlog_offset）
5. 旧 vlog 文件整体删除
```

**SSTable 指针更新方式**：

- 不重写整个 SSTable（代价过高）
- 引入 **indirection layer**：SSTable 的 value 指针改为 `vlog_id + vlog_offset + vlog_length`，通过 `vlog_id → 文件路径` 的 manifest 映射
- GC 完成后只更新 manifest（vlog_id 指向新文件），SSTable 中的 `vlog_id + offset` 不变
- 代价：读取多一次 manifest 查询（manifest 常驻内存）

**并发控制**：

| 阶段 | 新写入 | 读旧 vlog | 读新 vlog |
|------|--------|----------|----------|
| GC 期间 | 走活动 vlog（新文件） | 不受影响（旧文件仍存在） | 不适用 |
| GC 完成、删除旧 vlog 前 | 走活动 vlog | **需等待所有引用旧 vlog 的读完成** | 不适用 |
| 删除旧 vlog | 走活动 vlog | 拒绝（文件已删） | 走活动 vlog |

**安全屏障**：旧 vlog 删除前，等待 `active_iterators_` 计数器归零（复用现有 `CleanupDeletedFiles` 机制），确保没有迭代器还在读旧 vlog。

**降级与禁用**：

- `Options::vlog_gc_enabled = false` 时，vlog 只增不减，需人工清理
- 嵌入式场景可禁用 GC，定期重启重建 vlog（启动时从 SSTable 重写 vlog）

## 9. ZSet 索引优化 {#zset-index}

### 9.1 问题

当前所有 ZSet 操作（ZRANGE、ZRANK、ZCOUNT 等）都调用 `zscan_all` 进行全量 Scan + 内存排序，O(N log N)。

### 9.2 方案：引入内存跳表索引

```
写入路径：
1. ZADD key score member
2. 写入 KV 存储：score_key → member, member_key → score
3. **同时更新内存跳表索引**：score → {member}

读取路径：
1. ZRANGE key 0 -1
2. 从内存跳表直接遍历有序结果 → O(K) 而不是 O(N log N)
3. ZRANK key member → 跳表 rank 查询 → O(log N)
```

### 9.3 内存跳表生命周期

```cpp
// ZSet 索引节点：按 (score, member) 复合键排序
// 与 Redis 自身的 zskiplist 语义一致：
// - 主键 score 升序
// - 同 score 下按 member 字典序
struct ZSetNode {
    double score;
    std::string member;

    bool operator<(const ZSetNode& other) const {
        if (score != other.score) return score < other.score;
        return member < other.member;  // 同 score 按字典序
    }
};

class ZSetIndex {
    // 跳表以 ZSetNode 为复合键，value 位仅作占位（void）
    // ZRANGE 遍历跳表、ZRANK 用 rank 字段、ZSCORE 反向查 member→score
    using Index = SkipList<ZSetNode, void>;

    // key → 索引实例
    std::unordered_map<std::string, std::unique_ptr<Index>> indices_;
    // 反向表：member → score，用于 ZSCORE / ZRANK 快速定位
    std::unordered_map<std::string, std::unordered_map<std::string, double>> reverse_;

    void Rebuild(const std::string& key);  // 从 KV 存储重建索引
    // 触发时机：检测到 ZSet key 存在但索引缺失时
};
```

**为什么不用 `SkipList<double, std::set<std::string>>`**：

- 跳表节点会挂一个 `std::set`，每节点膨胀严重
- 相同 score 下多个 member 的**字典序**无法由 Key 比较得出，只能由 value 内部的 `std::set` 保证，违反跳表"有序由 Key 决定"的语义
- ZRANGE 拿到的是 `std::set` 引用而非单个 member，需要二次遍历
- 复合键 `(score, member)` 直接由 SkipList 的 Key 比较器排序，所有 ZSet 命令都是 O(log N) 或 O(K)

### 9.4 持久化

- 索引不直接持久化，启动时从 KV 存储重建
- 重建时只扫描 ZSet 前缀的 key，不影响其他数据类型
- 启动命令：重建在后台线程渐进式进行，不阻塞主线程

各语言 SDK 应提供**等价**的构造与命令接口，仅语法差异。下表给出三语言等价签名：

```python
# Python
class LightKVClient:
    def __init__(self,
                 hosts: list[str],          # 多节点地址
                 pool_size: int = 10,       # 连接池大小
                 timeout_ms: int = 3000,    # 超时
                 retry: int = 3):           # 重试次数
    # 所有标准 Redis 命令
    def set(self, key: str, value: str) -> bool: ...
    def get(self, key: str) -> str | None: ...
    def zrange(self, key: str, start: int, stop: int) -> list[str]: ...
```

```go
// Go
type LightKVClient struct {
    // ...
}
func NewLightKVClient(hosts []string, opts ...Option) (*LightKVClient, error)
// Option 模式：WithPoolSize / WithTimeout / WithRetry
func (c *LightKVClient) Set(key, value string) (bool, error)
func (c *LightKVClient) Get(key string) (string, error)  // 空字符串+ErrNil 表示不存在
func (c *LightKVClient) ZRange(key string, start, stop int) ([]string, error)
```

```javascript
// Node.js
class LightKVClient {
    constructor(hosts, { poolSize = 10, timeoutMs = 3000, retry = 3 } = {})
    async set(key, value)  // → boolean
    async get(key)         // → string | null
    async zrange(key, start, stop)  // → string[]
}
```

**等价约定**：

| 行为 | Python | Go | Node.js |
|------|--------|----|---------|
| key 不存在 | 返回 `None` | 返回 `""` + `ErrNil` | 返回 `null` |
| 命令失败 | 抛异常 | 返回 `error` | reject Promise |
| 超时 | 抛 `TimeoutError` | 返回 `context.DeadlineExceeded` | reject `TimeoutError` |
